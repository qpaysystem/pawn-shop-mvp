# Выгрузка 1С «Операции»: структура и маппинг для интеграции

На основании реальной выгрузки из 1С (отчёт по операциям залога и скупки) ниже описаны колонки, их смысл и предложенный маппинг на таблицы 1С и сущности приложения. Документ служит основой для настройки синхронизации и связей.

---

## 1. Структура отчёта 1С (колонки)

Выгрузка содержит строки двух типов операций в одном формате:

| № | Колонка в выгрузке | Пример значения | Смысл |
|---|--------------------|-----------------|--------|
| 1 | **Дата** | 04 марта, 11 марта | Дата операции (краткий формат). |
| 2 | **Тип операции + номер + дата/время** | Операция по залогу 00БП-004246 от 04.03.2026 14:58:12 | Тип документа, номер (00БП-XXXXXX), дата и время из 1С. |
| 3 | **Точка (адрес/наименование)** | Дуси Ковальчук 266/2, Карла Маркса 5/1, Горский, 1, Горский сейф | Склад/магазин/точка проведения операции. |
| 4 | **Клиент (ФИО)** | Макарова Елизавета Андреевна, Новиков Евгений Александрович | Контрагент — физлицо из справочника. |
| 5 | **Наименование товара** | Цепь, Золото 585, 8,67 гр.; Крест, Золото 585, 12,31 гр. | Описание позиции: тип изделия, проба, вес, иногда доп. текст. |
| 6 | **Сумма** | 50 000, 112 200, 1 045 000 | Сумма по строке (займ или скупка). |
| 7–8 | (пусто) | — | Резерв или доп. колонки в отчёте. |

**Два типа операций в одной выгрузке:**

- **Операция по залогу** — номер вида `00БП-004246`, `00БП-004840` (диапазон 4xxx и т.п.). Соответствует документу **Залог** в 1С (таблица документа залога — уточнить по метаданным, не _document389x1).
- **Скупка ценностей** — номер вида `00БП-000578`, `00БП-000622`. Соответствует документу **Скупка** в 1С, в БД — **`_document389x1`**.

Один номер документа может иметь несколько строк (несколько товаров в одном договоре).

## 3. Предполагаемый маппинг: отчёт → таблицы 1С → приложение

### 3.1. Общие связи

| Колонка отчёта | Источник в 1С (предположение) | Поле/сущность в приложении |
|----------------|------------------------------|----------------------------|
| Дата | Документ: `_date_time` или аналог | `pawn_contracts.loan_date` / `purchase_contracts.purchase_date` |
| Номер документа | Документ: `_number` или реквизит номера | Сопоставление по номеру; у нас `contract_number` (свой формат) и `lmb_doc_uid` (hex `_idrref`) |
| Точка | Ссылка на справочник складов/точек в документе | `pawn_contracts.store_id` / `purchase_contracts.store_id` → `stores.id` (маппинг по имени/адресу) |
| Клиент (ФИО) | Документ: реквизит-ссылка на контрагента (физлицо) | `clients.id` через `user_uid` = hex `_idrref` из `_reference122x1` |
| Наименование товара | Заголовок документа или ТЧ: текстовое поле (в скупке — _fld9638, _fld9643, _fld9650) | `items.name` (при синхронизации и по команде `lmb:refresh-purchase-item-names-from-1c`) |
| Сумма | Документ или ТЧ: сумма (займ/скупка) | `pawn_contracts.loan_amount` / `purchase_contracts.purchase_amount` |

### 3.2. Документ «Скупка ценностей» (_document389x1)

| Элемент | Таблица/колонка 1С | Конфиг / приложение |
|---------|--------------------|----------------------|
| Документ | `public._document389x1` | `LMB_1C_PURCHASE_DOCUMENT_TABLE`, команда переноса в скупку и обновление item.name |
| Уникальный ключ | `_idrref` (храним hex в `lmb_doc_uid`) | `purchase_contracts.lmb_doc_uid` |
| Контрагент | `_fld9626rref` → `_reference122x1._idrref` | `LMB_1C_PAWN_CONTRAGENT_COLUMN` (используется при синхронизации из 389x1) |
| Наименование товара (текст) | Заголовок: `_fld9638`, `_fld9643`, `_fld9650` (первая непустая) | `config/services.php` → `lmb_1c_purchase_item_name.name_columns` |
| Номер | реквизит номера (вид 00БП-000578) | По выгрузке метаданных уточнить имя колонки (`_number` или `_fld*`) |
| Дата | `_date_time` или аналог | По метаданным |
| Сумма | реквизит суммы в заголовке или в ТЧ | По метаданным (в скупке ТЧ пусты — сумма в заголовке) |
| Склад/точка | реквизит-ссылка на склад/организацию | Уточнить по метаданным; затем маппинг на `stores.id` |

Табличные части `_document389_vt9655` и `_document389_vt9675` в текущей базе пусты — описание товара и сумма берутся из **заголовка** документа.

### 3.3. Документ «Залог» (Операция по залогу)

| Элемент | Таблица/колонка 1С | Конфиг / приложение |
|---------|--------------------|----------------------|
| Документ | Таблица документа «Залог» — **ещё не определена** (не _document389x1) | В .env оставить `LMB_1C_PAWN_DOCUMENT_TABLE` пустым до уточнения по метаданным |
| Номер | Вид 00БП-004246 | После определения таблицы — реквизит номера |
| Контрагент | Аналогично: ссылка на _reference122x1 | Как в lmb_1c_pawn_sync |
| Наименование товара | ТЧ документа или реквизит заголовка | По метаданным; при наличии ТЧ — номенклатура или описание по строке |
| Сумма по строке | ТЧ или заголовок | `pawn_contracts.loan_amount` |

Чтобы однозначно определить таблицу залога в БД 1С: по выгрузке метаданных найти документ с синонимом «Залог» / «Операция по залогу» и зафиксировать физическое имя таблицы (например _document31784). Альтернатива: перебор `_document*` по полям (дата, номер, контрагент, сумма) и сопоставление с номерами 00БП-004xxx.

## 4. Связи между таблицами (схема для интеграции)

```
1С (PostgreSQL lmb)
─────────────────────────────────────────────────────────────────────
_reference122x1          Справочник контрагентов (физлица)
  _idrref (PK)     ──────┐
  _fld3178 (ФИО)         │
  _fld41084 (телефон)   │
  _fld3184, _fld3185    │
                        │
_document389x1          Документ «Скупка ценностей»
  _idrref (PK)          │
  _fld9626rref  ─────────┘ контрагент
  _fld9638, _fld9643, _fld9650  (наименование товара)
  _date_time, _number, сумма...
  (_document389_vt* — ТЧ пусты)

_document????            Документ «Залог» (таблица уточняется)
  _idrref (PK)
  _fld????rref  ───────────────→ _reference122x1._idrref
  _number (00БП-004xxx), _date_time, сумма...
  _document????_vt*      ТЧ: номенклатура/описание, сумма по строке

Справочник складов/точек (таблица уточняется по метаданным)
  _idrref  ←──────────── документы (склад/точка в реквизите)

─────────────────────────────────────────────────────────────────────
Приложение (Laravel)
─────────────────────────────────────────────────────────────────────
clients
  user_uid (hex)  ←────── encode(_reference122x1._idrref, 'hex')

stores
  name, address    ←────── маппинг по наименованию точки из отчёта / справочника 1С

items
  name             ←────── из _fld9638/_fld9643/_fld9650 (скупка) или из ТЧ залога
  lmb_ref (hex)    ←────── при наличии номенклатуры в 1С: ссылка на справочник номенклатуры

purchase_contracts
  lmb_doc_uid      ←────── encode(_document389x1._idrref, 'hex')
  client_id        ←────── по user_uid
  item_id          ←────── создаётся/обновляется при синхронизации
  store_id         ←────── по маппингу точка 1С → stores.id
  purchase_amount, purchase_date

pawn_contracts
  lmb_doc_uid      ←────── encode(_documentЗалог._idrref, 'hex') после определения таблицы
  client_id, item_id, store_id, loan_amount, loan_date ...
```

## 5. Что настроить для интеграции

1. **Метаданные 1С**  
   Выгрузить структуру (обработка из `docs/1c_export_metadata_to_file.txt`) и зафиксировать:
   - имя документа «Залог» и физическую таблицу (`_document*`);
   - реквизиты документов Залог и Скупка: номер, дата, контрагент, сумма, **склад/точка**, в ТЧ — номенклатура/описание, сумма по строке;
   - справочник складов/точек (имя и таблица).

2. **Конфиг приложения**  
   - Для скупки уже есть: `lmb_1c_purchase_item_name` (document_table, name_columns).  
   - Для залога: после определения таблицы задать `LMB_1C_PAWN_DOCUMENT_TABLE`, колонки контрагента, даты, номера, суммы, ТЧ (если есть).  
   - При появлении колонки «склад» в документе — добавить в конфиг и маппинг 1С «точка» → `stores.id` (по имени или через справочник в 1С).

3. **Маппинг точек**  
   Создать в приложении или в конфиге соответствие: «Дуси Ковальчук 266/2» → store_id, «Горский, 1» → store_id и т.д., чтобы при синхронизации подставлять `store_id` в договоры.

4. **Номенклатура**  
   Если в 1С по залогу/скупке будет найдена ссылка на справочник номенклатуры — хранить в `items.lmb_ref` и при синхронизации подтягивать наименование из справочника; иначе продолжать использовать текстовые поля документа (как сейчас для скупки).
