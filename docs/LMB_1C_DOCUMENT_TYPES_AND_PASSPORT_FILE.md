# Виды документов удостоверяющих личность залогодателя и файл паспорта в БД 1С

Документ фиксирует **находки по PostgreSQL** (база ломбарда): где лежат **виды документа**, **скан паспорта**, как не путать ссылки.

---

## 1. Файл / скан паспорта (запомнено для разбора)

У контрагента в **`public._reference122x1`** может быть реквизит-ссылка **`_fld41630rref`** (bytea).

- Она указывает на запись в **`public._reference362x1`** (справочник вложений / файлов в этой конфигурации).
- В **`_reference362x1`** для такой записи видны, например: **`_description`** = имя файла (`IMG_....jpg`), **`_fld8241`** = расширение, **`_fld8240`** = размер — это **метаданные файла**, не серия/номер паспорта в виде текста.
- Текстовые поля вложения (**`_fld8237`**, **`_fld8239`**, **`_fld41761`**) в проверенном примере были **пусты** (нет OCR в БД).

**Поиск таблицы по hex-ссылке** (какой справочник скрывается за `*rref`):

```bash
php scripts/find_idrref_in_1c.php <32_символа_hex_без_0x>
```

Итог: «паспорт как файл» в SQL = **ссылка на `_reference362x1`**, а не колонки `_fld3201`/`_fld3202`.

## 2. Все варианты «вида документа» в базе (перечисление)

Вид документа удостоверяющего личность в хранилище **`public._reference122_vt3220x1`** задаётся ссылкой **`_fld3222rref`**.

По проверке в этой базе значения **`_fld3222rref`** совпадают с **`_idrref`** в перечислении:

### Таблица `public._enum1039`

| `_enumorder` | `encode(_idrref,'hex')` | Примечание |
|--------------|--------------------------|------------|
| 0 | `8c76e5a8c09bc9114e49e36a717ac118` | Чаще всего в данных |
| 1 | `a95fd3e53f7b10fb46f57b9225f4b5fe` | Второй по частоте |
| 2 | `b0095c4caf46fb50471b8bc8007ddfa8` | |
| 3 | `8161148761f296984ad9ddfdfef27478` | |
| 4 | `95c9b088b8e438d04864721afbbea4fa` | |
| 5 | `acb548e3310c1b5345a462ffc8a18d99` | |
| 6 | `bbccefae8ad16d254350fd4595461b46` | |

**Всего значений в перечислении:** 7.

**Важно:** в PostgreSQL в **`_enum1039`** обычно **нет текстовых названий** («Паспорт РФ», «Водительское удостоверение» и т.д.) — только **`_idrref`** и **`_enumorder`**. Расшифровку даёт **Конфигуратор 1С** (перечисление с внутренним ID, соответствующим этой таблице) или выгрузка метаданных.

### Как узнать, **какой вид документа у каждого hex** (русское название)

**Из одной только базы PostgreSQL — нельзя:** таблица `_enum1039` не содержит строк с подписями значений.

Сделайте одно из следующего:

1. **Конфигуратор 1С** (рекомендуется)  
   - Откройте дерево метаданных → **Перечисления**.  
   - Найдите перечисление, которое используется как «вид документа удостоверяющего личность» / составной тип контактной информации (точное имя в конфигурации «Рарус: Ломбард» может называться иначе — ищите по синониму «документ», «удостоверяющий»).  
   - Откройте список **значений перечисления**: порядок **`_enumorder`** в БД обычно совпадает с порядком строк в дереве (0, 1, 2, …).  
   - Сопоставьте: **порядок в форме** ↔ **`_enumorder`** в таблице выше ↔ **hex** из колонки.

2. **Выгрузка конфигурации в файлы (XML)**  
   - В XML ищите узлы перечисления и значений; иногда рядом с внутренними идентификаторами есть **GUID/ссылка**, похожая на ваш hex (можно искать подстроку `8c76e5a8-c09b-...` в формате UUID).

3. **Эмпирически по данным**  
   - Для двух основных hex, которые массово встречаются в `vt3220` (`8c76…`, `a95f…`), в 1С откройте несколько карточек контрагентов с разными видами документа и посмотрите, какой hex в БД у какого отображаемого вида (сравнение `encode(_fld3222rref,'hex')` с полем на форме).

После того как названия сняты с конфигурации, их можно **вручную занести в справочник-мэппинг** в приложении (hex → строка для UI).

### Фактическое использование в `vt3220` (пример подсчёта по `_fld3222rref`)

В данных встречаются в основном:

- `8c76e5a8c09bc9114e49e36a717ac118` — наибольшее число строк;
- `a95fd3e53f7b10fb46f57b9225f4b5fe` — реже;
- нулевая ссылка `00000000000000000000000000000000` — единичные строки (как «не задано»).

Остальные 5 значений перечисления **могут быть в справочнике**, но **редко или не использоваться** в текущих строках `vt3220` — при необходимости пересчитать:

```sql
SELECT encode(vt."_fld3222rref", 'hex') AS doc_type_hex, COUNT(*)::bigint
FROM public._reference122_vt3220x1 vt
WHERE vt."_fld3222rref" IS NOT NULL AND octet_length(vt."_fld3222rref") > 0
GROUP BY 1
ORDER BY 2 DESC;
```

## 4. Обновление при смене базы

Номера таблиц (`_enum1039`, `_reference362x1`, поле `_fld41630rref`) привязаны к **конкретной выгрузке** конфигурации. После смены базы или обновления 1С перезапустите запросы из этого файла и при необходимости обновите таблицу hex-значений.
