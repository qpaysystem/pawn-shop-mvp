# Метаданные конфигурации 1С «Ломбард» (редакция 4.0)

Документ описывает **логическую структуру** метаданных конфигурации «1С-Рарус: Ломбард», извлечённую из архива «МетаданныеЛомбард» (HTML-выгрузка). Используется для улучшения интеграции с нашей БД: сопоставление логических имён объектов 1С с физическими таблицами PostgreSQL (`_referenceNNN`, `_documentNNN`) и настройка маппинга полей.

**Источник:** файл из архива (например `МетаданныеЛомбард.htm`). Версия конфигурации в выгрузке: 4.0.89.2.

## 4. Как узнать физическую таблицу (PostgreSQL) по логическому имени

Метаданные в HTML **не содержат** внутренних ID объектов 1С (номера `NNN` в `_referenceNNN`). Чтобы получить соответствие «РС_ИсточникРекламы» → `_referenceXXX`:

1. Выполнить на БД 1С (PostgreSQL) команду просмотра схемы:  
   `php artisan lmb:db-tables`  
   (или аналог — вывод списка таблиц и, при наличии, комментариев/описаний).
2. Найти таблицу справочника, в которой хранятся элементы «Источник рекламы» (по названию полей или по ссылкам из `_reference122x1._fld41085rref`).
3. Задать в `.env`:  
   `LMB_1C_TRAFFIC_SOURCE_REF_TABLE=_referenceXXX`  
   (подставив найденное имя таблицы).

После этого синхронизация контрагентов сможет подставлять название источника рекламы и записывать его в `traffic_sources` / `clients.traffic_source_id`.

## 5. Краткая сводка для конфига

- **Контрагенты (физлица):** таблица `_reference122x1`. Дата создания: колонка из `date_created_column`. Источник рекламы: колонка `_fld41085rref`, таблица справочника — из `LMB_1C_TRAFFIC_SOURCE_REF_TABLE` (логическое имя справочника в 1С: **РС_ИсточникРекламы**).
- **Склады (подразделения/филиалы):** в текущей интеграции используются из `_reference197` (подразделения); в метаданных им соответствуют справочники **ПодразделенияОрганизаций** и **ЛМБ_Филиалы** (в документах тип «Филиал» — ЛМБ_Филиалы, «Склад» может быть другим справочником).
- **Залог (операции):** документ **ЛМБ_ОперацияПоЗалогу**. Номер физической таблицы — из выгрузки схемы БД.
- **Скупка:** документ **ЛМБ_СкупкаЦенностей**, склад в реквизите **Кладовая**.

Дополнительная структура таблиц и колонок — в `docs/LMB_1C_TABLES_STRUCTURE_AND_SYNC.md`.

В блоке **Контрагенты (Контрагенты)** перечислены реквизиты, в т.ч.:

Тип **ДокументУдостоверяющийЛичность** в этой выгрузке указан как **Строка(250)** — т.е. в конфигурации это один строковый реквизит, а не составной тип «Паспорт РФ» с отдельными полями Серия/Номер в той же таблице.

### 6.3. Результат проверки по SQL для контрагента Фромич Анна Владимировна

Для записи с `encode(_idrref, 'hex') = '812a0050569bbedf11eaa55bd5c03527'` в таблице `public._reference122x1`:

- **Совпадает с формой:**  
  `_fld3178` = «Фромич Анна Владимировна», `_fld41084` = «79612209950», `_fld3191` = 1989-01-24 (дата рождения).
- **Паспорт в колонках серия/номер:**  
  `_fld3184`, `_fld3185`, `_fld3201`, `_fld3202` — **все пустые** (длина 0 или 1 символ).

То есть те поля формы «Серия» и «Номер» паспорта, которые на скриншоте заполнены (50 19, 961613), **в этой базе не попадают** в колонки `_fld3201`/`_fld3202` (и не в `_fld3184`/`_fld3185`) для данного контрагента. Данные с формы могут сохраняться:

- в другом реквизите той же таблицы (например, один столбец «Документ» типа строка 250, если платформа так выгружает);
- в хранилище значений или в связанной структуре по одной из ссылок `_fld*rref`;
- или форма при открытии читает паспорт из иного места (регистр, подчинённая таблица), а не из `_reference122x1._fld3201`/`_fld3202`.

### 6.4. Вывод для синхронизации паспорта

- Текущая синхронизация паспорта из 1С использует `passport_series_column` = `_fld3202` и `passport_number_column` = `_fld3201`. Для контрагентов, у которых на форме указаны серия и номер паспорта, но в БД эти колонки пусты, синхронизация паспорта из 1С их не подтянет.
- Чтобы подтягивать паспорт для таких записей, нужно в Конфигураторе 1С или через выгрузку метаданных/схемы БД установить, **в каком столбце или таблице** реально хранятся «Серия» и «Номер» (или одна строка «Документ»), и при необходимости добавить в конфиг другие имена колонок или отдельный запрос к хранилищу значений/связанной таблице.

## 7. Поле «Документ удостоверяющий личность залогодателя»

**Залогодатель** в документах залога (ЛМБ_ОперацияПоЗалогу) — это **Контрагент** (справочник Контрагенты). Документ удостоверяющий личность залогодателя = реквизит **ДокументУдостоверяющийЛичность** этого справочника.

### Где хранится в БД 1С

| Место | Таблица / колонки | Назначение |
|-------|-------------------|------------|
| **Расширение справочника Контрагенты** | `public._reference122x1` | Основная таблица по контрагенту (залогодателю). |
| Серия паспорта | `_fld3202` (основной), `_fld3184` (альт.) | Серия. |
| Номер паспорта | `_fld3201` (основной), `_fld3185` (альт.) | Номер. |
| Кем выдан | `_fld3197` | Текст «кем выдан» (в части баз пустой или используется под заметки). |
| **Хранилище значений типа 3220** | `public._reference122_vt3220x1` | Составной тип «Документ удостоверяющий личность» и контактная информация. Связь: колонка `_reference122_idrref` = `_idrref` контрагента в `_reference122x1`. В строках хранилища — XML/JSON с паспортом, в т.ч. текст «ГУ МВД…», «выдан» (колонки `_fld3224`, `_fld3225`, `_fld3227`, `_fld34009` и др.). |
| **Вид документа (ссылка)** | `_reference122_vt3220x1._fld3222rref` | По проверке по БД: значение ссылки совпадает с `_idrref` в перечислении **`public._enum1039`** (в этой базе). Текстовые названия значений перечисления в PostgreSQL в таблице `_enum1039` не хранятся — только `_idrref` и `_enumorder`; расшифровку («Паспорт гражданина РФ» и т.д.) смотреть в Конфигураторе 1С или по метаданным. **Полный список hex-значений и статистика использования:** **docs/LMB_1C_DOCUMENT_TYPES_AND_PASSPORT_FILE.md**. |
| Другой компонент типа | `_fld3223rref` | В выборке по `_enum*` совпадений с `_idrref` не найдено — возможно, ссылка на справочник (`_reference*`) или иной тип; см. `php artisan lmb:doc-identity-rref-lookup`. |
| **Файл / скан паспорта** | `_reference122x1._fld41630rref` → **`_reference362x1`** | Ссылка на вложение (фото/скан): метаданные файла в `_reference362x1` (`_description`, расширение, размер). Подробно: **docs/LMB_1C_DOCUMENT_TYPES_AND_PASSPORT_FILE.md**. |

В таблице `_reference122x1` **нет** колонки `_fld3220rref` — ссылки на хранилище из основной таблицы нет; связь односторонняя: в `_reference122_vt3220x1` у каждой строки указан владелец `_reference122_idrref`. Отдельной колонки «Строка(250)» с полным текстом документа в `_reference122x1` не выявлено (проверенная колонка `_fld3196` содержит иные данные).

**Итог:** документ удостоверяющий личность залогодателя в БД 1С задаётся:
1. В **`_reference122x1`**: серия/номер — `_fld3202`, `_fld3201` (или `_fld3184`, `_fld3185`), при необходимости «кем выдан» — `_fld3197`.
2. В **`_reference122_vt3220x1`**: строки с `_reference122_idrref` = контрагент, в которых в полях хранится паспорт/«МВД»/«выдан» (см. команду `lmb:export-passport-from-1c --source=vt3220`).
