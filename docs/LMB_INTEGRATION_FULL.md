# Полная интеграция с 1С Ломбард: договоры залога, залоговые вещи, скупка

Документ описывает порядок настройки и синхронизации данных между нашей базой (Laravel) и базой 1С (PostgreSQL): **клиенты**, **договоры залога**, **залоговые вещи**, **договоры скупки** и **вещи скупленные**.

---

## 1. Схема данных

| Наша сущность       | Источник в 1С                    | Таблица 1С (пример)     | Команда синхронизации        |
|---------------------|----------------------------------|-------------------------|------------------------------|
| **Клиенты**         | Справочник контрагентов (физлица) | `_reference122x1`       | `lmb:sync-contragents`       |
| **Договоры залога** | Документ «Операция по залогу»    | `_documentXXXXx1`      | `lmb:sync-pawn-contracts`    |
| **Залоговые вещи**  | Табличная часть документа залога | `_documentXXXX_vtYYYY` | в составе sync-pawn-contracts |
| **Договоры скупки** | Документ «Скупка ценностей»      | `_document389x1`        | `lmb:sync-purchase-contracts`|
| **Вещи скупленные** | То же (одна строка = один договор) | `_document389x1`       | в составе sync-purchase-contracts |

- **Залоговые вещи** в нашей базе — это записи в таблице `items`, связанные с `pawn_contracts` (один договор залога может иметь одну или несколько вещей по табличной части 1С).
- **Вещи скупленные** — записи в `items`, связанные с `purchase_contracts` (один договор скупки = один товар в 1С).

---

## 2. Порядок настройки (по шагам)

### Шаг 1. Подключение к БД 1С

В `.env` задайте:

```env
LMB_DB_DRIVER=pgsql
LMB_DB_HOST=192.168.7.250
LMB_DB_PORT=5432
LMB_DB_DATABASE=lmb
LMB_DB_USERNAME=lmb
LMB_DB_PASSWORD=...
```

Проверка: `php artisan lmb:db-schema --limit=5`

### Шаг 2. Синхронизация клиентов (контрагентов)

Клиенты должны быть в нашей базе, чтобы при синхронизации залогов и скупки подставлялся `client_id`.

```bash
php artisan lmb:sync-contragents
```

Конфиг уже задан: таблица `_reference122x1`, колонки телефона и ФИО. После синхронизации в разделе «Клиенты» появятся физлица из 1С.

### Шаг 3. Синхронизация договоров скупки и вещей скупленных

Таблица документа скупки уже известна: `_document389x1`. По умолчанию в конфиге указана она.

```bash
php artisan lmb:sync-purchase-contracts --dry-run   # проверка
php artisan lmb:sync-purchase-contracts             # запись в БД
```

В результате создаются/обновляются записи в `purchase_contracts` и `items` (вещи скупленные). Связь: один договор скупки = один товар (item).

### Шаг 4. Найти таблицу документа залога в 1С

Документ «Операция по залогу» (ЛМБ_ОперацияПоЗалогу) в PostgreSQL имеет вид `_documentNNN` или `_documentNNNx1`. **В части баз 1С таблица без суффикса x1** (например, только `_document31784`). Команда `lmb:db-schema --table=_document31784x1` при отсутствии такой таблицы автоматически покажет структуру `_document31784`. Номер NNN у каждой базы свой.

```bash
php artisan lmb:discover-1c-documents --count --exclude-known --with-vt
```

Команда выведет список таблиц `_document*` (кроме известной скупки), количество записей и подсказку по колонкам. Ищите таблицу с большим числом записей и колонками: дата (`_date_time`), номер (`_number`), ссылка на контрагента (`_fld*rref`), сумма. Для этой же таблицы будут выведены **табличные части** (`_document*_vt*`) — это залоговые вещи (номенклатура по строкам).

Пример вывода:

```
public._document31784x1
  Записей (без пометки): 1200
  Подсказка: похоже на документ с датой, номером и ссылкой (подходит для залога/операции).
  Табличная часть: public._document31784_vt31892
```

Запомните имена: например, документ `_document31784` (или `_document31784x1`, если есть), табличная часть `_document31784_vt31892`. В выводах discover все таблицы с 0 записей — это нормально; залоги могут быть в другой таблице или пока отсутствовать. Если записей нет, синхронизация просто не создаст договоров.

### Шаг 5. Настроить синхронизацию залогов в .env

Для конфигурации Ломбард 4.0 с документом **\_document31784** и табличной частью **\_document31784_vt31892** в конфиге уже заданы значения по умолчанию. В `.env` достаточно указать таблицу документа (в базах без x1 — именно `_document31784`):

```env
# Документ залога (ЛМБ_ОперацияПоЗалогу). В вашей базе таблица без x1.
LMB_1C_PAWN_DOCUMENT_TABLE=_document31784
# Остальные колонки заданы в config/services.php по умолчанию; при необходимости переопределите:
# LMB_1C_PAWN_CONTRAGENT_COLUMN=_fld31884rref
# LMB_1C_PAWN_AMOUNT_COLUMN=_fld31887
# LMB_1C_PAWN_PERCENT_COLUMN=_fld31888
# LMB_1C_PAWN_TABLE_PART_TABLE=_document31784_vt31892
# LMB_1C_PAWN_VT_DOC_ID_COLUMN=_document31784_idrref
# LMB_1C_PAWN_VT_NOMENCLATURE_COLUMN=_fld31894rref
# LMB_1C_PAWN_VT_AMOUNT_COLUMN=_fld31896
# LMB_1C_PAWN_VT_DESCRIPTION_COLUMN=_fld31895
LMB_1C_PAWN_DEFAULT_STORE_ID=1
```

Проверить структуру (для таблицы без x1 подставится корректное имя):

```bash
php artisan lmb:db-schema --table=_document31784
php artisan lmb:db-schema --table=_document31784_vt31892
```

Колонки заголовка документа (сумма займа, срок, сумма выкупа) при отличии структуры уточните по выводу первой команды и задайте в .env.

### Шаг 6. Запуск синхронизации залогов

```bash
php artisan lmb:sync-pawn-contracts --dry-run   # проверка
php artisan lmb:sync-pawn-contracts             # только действующие (expiry >= сегодня)
php artisan lmb:sync-pawn-contracts --all        # все залоги
```

В результате создаются/обновляются записи в `pawn_contracts` и `items` (залоговые вещи). По каждой строке табличной части создаётся отдельный договор залога с привязкой к одному item (или один договор на документ, если табличная часть не задана/пуста).

## 3. Итоговая схема: что где лежит

- **clients** — контрагенты из 1С (физлица), ключ связи `user_uid` = `encode(_idrref,'hex')` из `_reference122x1`.
- **pawn_contracts** — договоры залога из документа «Операция по залогу»; `lmb_doc_uid` = уникальный ID документа (и при необходимости номер строки ТЧ) в 1С; связь с клиентом по `client_id`, с вещью по `item_id`.
- **items** — и залоговые вещи, и вещи скупленные: различие по связям — у item есть либо `pawnContract`, либо `purchaseContract` (или оба не заданы, если товар создан вручную).
- **purchase_contracts** — договоры скупки из документа «Скупка ценностей»; `lmb_doc_uid` = `encode(_idrref,'hex')` из `_document389x1`; связь с клиентом и item.

## 4. Полезные команды

| Действие | Команда |
|----------|---------|
| Список таблиц документов 1С | `php artisan lmb:discover-1c-documents --count --exclude-known --with-vt` |
| Структура таблицы 1С | `php artisan lmb:db-schema --table=имя_таблицы` |
| Синхронизация клиентов | `php artisan lmb:sync-contragents` |
| Синхронизация скупки | `php artisan lmb:sync-purchase-contracts` |
| Синхронизация залогов | `php artisan lmb:sync-pawn-contracts` |
| Обновление наименований товаров скупки из 1С | `php artisan lmb:refresh-purchase-item-names-from-1c` |
