# Справочники 1С LMB: структура и маппинг на нашу базу

Обзор справочников (каталогов) в БД 1С и соответствие сущностям приложения. Определено по количеству записей, структуре таблиц и примерам данных (без выгрузки метаданных имена «Контрагенты», «Филиалы» и т.д. — предположительные).

---

## 1. Клиенты (залогодатели / контрагенты — физлица)

| Параметр | Значение |
|----------|----------|
| Таблица | `_reference122x1` |
| Записей | ~23 650 |
| Назначение | Физические лица: ФИО, телефон, паспорт. |

**Ключевые колонки:** `_idrref`, `_description`, `_fld3178` (ФИО), `_fld41084` (телефон), `_fld3184`/`_fld3185` (паспорт).

**Маппинг на приложение:** `clients`. Связь: `clients.user_uid` = `encode(_reference122x1._idrref, 'hex')`. Синхронизация: `php artisan lmb:sync-contragents`. Конфиг: `config/services.php` → `lmb_1c_contragent_sync`.

**Контрагенты — юр. лица/банки:** `_reference108` (~1 390). Для клиентов ломбарда не используем.

## 2. Филиалы (подразделения / точки)

| Параметр | Значение |
|----------|----------|
| Таблица | `_reference197` |
| Записей | 65 |
| Назначение | Подразделения и точки: «Дуси Ковальчук 266/2», «Горский, 1», «Вокзал ювелирка», «Титова 30», «Мичурина, 23/1», «Станиславского, 29», «Колхидская, 11» и т.д. |

**Ключевые колонки:** `_idrref`, `_code` (например БП-000008, 00-000002), `_description` (наименование), `_fld5205` (mvarchar 250 — возможно адрес), `_fld40540`, `_fld40545` (mvarchar 254).

**Маппинг на приложение:** `stores`. Связь: `stores.lmb_store_uid` = `encode(_reference197._idrref, 'hex')`. В документах залога/скупки склад/филиал задаётся ссылкой на этот справочник; маппинг по `lmb_store_uid` или `LMB_1C_PAWN_STORE_MAPPING` в .env.

**Команды:** список кодов филиалов из документа залога: `php artisan lmb:list-pawn-store-refs`. Синхронизация филиалов из 1С в `stores`: `php artisan lmb:sync-stores-from-1c` (конфиг: `config/services.php` → `lmb_1c_stores_sync`, таблица по умолчанию `_reference197`).

## 3. Сотрудники

| Параметр | Значение |
|----------|----------|
| Таблица | `_reference121` |
| Записей | 4 (в данной базе) |
| Назначение | Сотрудники: код БП-000001, наименование — ФИО (например «Зайцев Дмитрий Михайлович»). |

**Ключевые колонки:** `_idrref`, `_code`, `_description`. Расширенной таблицы `_reference121x1` в списке нет — реквизиты могут быть в основной.

**Маппинг на приложение:** `users` (сотрудники системы) или отдельная таблица «Сотрудники 1С». Пока синхронизация сотрудников из 1С не реализована; при необходимости связь по `encode(_idrref, 'hex')` в поле типа `user.lmb_user_uid` или в отдельной таблице.

## 4. Номенклатура

| Параметр | Значение |
|----------|----------|
| Таблица | `_reference140x1` |
| Записей | ~90 934 |
| Назначение | Номенклатура товаров/услуг: иерархия групп и элементы. Группы: «Ломбард», «Скупка золота», «Услуги», «Ювелирные изделия», «Залоговое имущество», «Техника», «Шубы», «Строительный инструмент», «Скупка», «Комплект», «Прочее» и т.д. |

**Ключевые колонки:** `_idrref`, `_code`, `_description`, `_parentidrref` (родитель в иерархии), `_folder` (признак группы). Реквизиты: `_fld3681` (mvarchar 25), `_fld3682rref` … `_fld3698` — ссылки на подсправочники (вид, марка, модель, металл и т.д.; точное назначение по метаданным).

**Маппинг на приложение:**  
- Группы номенклатуры → `item_categories` (дерево по `parent_id`).  
- Элементы номенклатуры (товары) → при создании залога/скупки из 1С заполняются `items.name`, `items.lmb_ref` = hex ссылки на номенклатуру; категория и бренд — при наличии маппинга подсправочников 1С на `item_categories` и `brands`.

---

## 5. Подсправочники номенклатуры (виды, марки, модели, драгметаллы)

В 1С номенклатура часто имеет реквизиты-ссылки на справочники «Вид номенклатуры», «Марка», «Модель», «Драгоценный металл» / «Проба». В `_reference140x1` есть колонки `_fld3682rref` … `_fld3698` (и др.) — это ссылки на другие справочники. Без выгрузки метаданных 1С однозначно сопоставить каждую колонку с «марка» или «металл» нельзя.

**Рекомендация:** выгрузить метаданные (см. `docs/1c_export_metadata_to_file.txt`) и в Конфигураторе найти справочники, подчинённые номенклатуре (или входящие в конфигурацию ЛМБ с именами типа «Вид номенклатуры», «Марки», «Проба металла»). Затем по количеству записей сопоставить с таблицами `_reference*` (например справочники на 10–100 записей с короткими наименованиями «585», «Золото», «Серебро» — кандидаты под металлы/пробы).

**Маппинг на приложение:**  
- Виды номенклатуры → `item_categories`.  
- Марки → `brands`.  
- Модели → при необходимости отдельное поле или справочник.  
- Драгоценные металлы/пробы → при необходимости поля в `items` (например `metal`, `sample`) или справочник.

## 6. Команды и следующий шаг

| Действие | Команда / конфиг |
|----------|-------------------|
| Список справочников с количеством записей | `php artisan lmb:db-tables --prefix=_reference --no-vt --count` |
| Схема конкретного справочника | `php artisan lmb:db-schema --table=_reference197` |
| Коды складов из документа залога | `php artisan lmb:list-pawn-store-refs` (для маппинга филиалов → stores) |
| Синхронизация клиентов | `php artisan lmb:sync-contragents` |
| Список справочников с примерами | `php artisan lmb:list-reference-catalogs` (опции: `--table=`, `--limit=`, `--sample=`) |
| Синхронизация филиалов в stores | `php artisan lmb:sync-stores-from-1c` (опции: `--dry-run`, `--force`) |

**Принципы для нашей базы:**  
- Использовать те же сущности, что и в 1С: клиенты, филиалы (магазины), сотрудники, номенклатура (категории + товары).  
- Связь с 1С по уникальному коду ссылки: `encode(_idrref, 'hex')` хранить в полях вида `*_uid` или `lmb_*_ref`.  
- Справочники с малым числом записей (филиалы, сотрудники, виды/марки/металлы) можно синхронизировать целиком; номенклатуру — выборочно или по мере появления в документах залога/скупки.
