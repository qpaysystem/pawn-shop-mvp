# Анализ структуры БД 1С в PostgreSQL

Практическое руководство: как **продолжать разбирать** хранение данных в базе ломбарда (1С → PostgreSQL), не имея прямого доступа к Конфигуратору для каждого поля.

Связанные документы:

- **docs/LMB_1C_CONTRAGENT_FULL_DATA.md** — залогодатель / контрагент: ФИО, телефон, паспорт, вид документа, прописка, пример SQL.
- **docs/LMB_1C_DOCUMENT_TYPES_AND_PASSPORT_FILE.md** — перечисление видов документа (`_enum1039`), скан паспорта (`_reference362x1`).
- **docs/LMB_1C_TABLES_STRUCTURE_AND_SYNC.md** — именование таблиц, общие колонки, уже известные маппинги (контрагенты, залоги и т.д.).
- **docs/LMB_1C_METADATA_LOMBARD.md** — логические имена объектов из выгрузки метаданных.
- **docs/LMB_1C_STRUCTURE_AND_API.md** — как вести заметки по структуре из Конфигуратора.

## 1. Общая логика платформы 1С в SQL

| Что в 1С | Что в PostgreSQL |
|----------|------------------|
| Справочник | `_referenceNNN` (+ часто `_referenceNNNx1` с реквизитами) |
| Табличная часть справочника / хранилище значений | `_referenceNNN_vtMMMM`, `_referenceNNN_vtMMMMx1` |
| Документ | `_documentNNN` (+ `x1`, табличные части `_vt`) |
| Перечисление | `_enumNNN` (в таблице обычно только `_idrref`, `_enumorder`; **тексты значений в SQL часто отсутствуют**) |
| Регистр сведений | `_infor*`, `_inforeg*` |
| Регистр накопления | `_accum*` |
| Ссылка на объект | Колонка `*rref` (bytea) = значение `_idrref` целевой таблицы |

Числа `NNN` — **внутренние ID** метаданных; без выгрузки конфигурации имя «Контрагенты» ↔ `_reference122` нужно находить эмпирически (по данным, документации проекта, Конфигуратору).

## 2. Быстрый обзор базы (одна команда)

```bash
php artisan lmb:postgres-structure-overview
php artisan lmb:postgres-structure-overview --top=25 --columns
```

Вывод:

- сколько таблиц в каждой **категории** (справочники, документы, `_vt`, перечисления, регистры и т.д.);
- **крупнейшие таблицы** по размеру на диске и оценке числа строк (`n_live_tup`);
- опционально — сколько таблиц содержат `_idrref` / `_marked`.

Это даёт ориентир: куда смотреть дальше и какие объёмы данных ожидать.

## 3. Пошаговый разбор конкретной сущности

1. **Список таблиц по префиксу** (не перечислять всю базу вручную):

   ```bash
   php artisan lmb:db-tables --prefix=_reference --no-vt --limit=0
   php artisan lmb:db-tables --prefix=_document --limit=0
   ```

2. **Структура таблицы** (все колонки и типы):

   ```bash
   php artisan lmb:db-schema --table=_reference122x1
   ```

3. **Поиск смысла колонки**  
   - по **данным**: `SELECT` с `encode(_idrref,'hex')`, типичные `_fld*`, ограничение `WHERE NOT _marked` и `LIMIT 20`;  
   - по **ссылке** `*rref`: сравнить значение с `_idrref` в кандидатах (`_enum*`, `_reference*`) — см. `php artisan lmb:doc-identity-rref-lookup` для примера с документом удостоверяющим личность.

4. **Табличные части и хранилища**  
   Имя вида `_reference122_vt3220x1` означает: у справочника 122 есть подчинённая структура / значение составного типа с внутренним номером `3220` в метаданных.

5. **Документы залога / скупки**  
   Используйте уже имеющиеся команды обнаружения:  
   `lmb:discover-1c-documents`, `lmb:pawn-document-sample`, см. **docs/LMB_1C_TABLES_STRUCTURE_AND_SYNC.md**.

## 4. Полезные команды (шпаргалка)

| Задача | Команда |
|--------|---------|
| Обзор категорий и крупных таблиц | `lmb:postgres-structure-overview` |
| Список таблиц по маске | `lmb:db-tables --prefix=_reference` |
| Колонки одной таблицы | `lmb:db-schema --table=ИМЯ` |
| Таблицы с «3220» (документ удостоверяющий личность) | `lmb:doc-identity-tables --schema` |
| Куда ссылаются rref в vt3220 | `lmb:doc-identity-rref-lookup` |
| Поиск паспортных полей по тексту | `lmb:find-passport-in-1c` |

## 5. Что фиксировать в документации проекта

При нахождении нового соответствия «логика 1С → таблица/колонка»:

1. Добавить строку в **docs/LMB_1C_TABLES_STRUCTURE_AND_SYNC.md** или узкоспециализированный файл (как **LMB_1C_PASSPORT_TABLES_SEARCH.md**).
2. При необходимости — параметр в **config/services.php** / `.env`.
3. Указать **ограничения**: например, перечисления без текстов в SQL, только `encode(_idrref,'hex')`.

Так структура хранения данных остаётся воспроизводимой для команды и для следующих сессий разработки.
