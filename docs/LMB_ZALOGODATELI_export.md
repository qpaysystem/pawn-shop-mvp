# Получение списка действующих залогодателей с личными и контактными данными

## Текущая ситуация

1. **PostgreSQL (192.168.7.250, БД lmb, пользователь lmb)**  
   - Подключение есть, структура таблиц видна (команда `php artisan lmb:db-schema`).  
   - **Прав на чтение данных нет:** при `SELECT` из таблиц 1С (_acc38, schemastorage и др.) возвращается ошибка **«нет доступа к таблице»**.  
   - Поэтому выборка залогодателей напрямую из БД с текущей ролью **невозможна**.

2. **HTTP API 1С (LmbUserApiService)**  
   - Есть метод **контрагент по телефону**: `GET .../user/{phone}`.  
   - Метода **список залогодателей** (или контрагентов с залогами) в API **нет**.

---

## Варианты решения

### Вариант A: Новый метод в HTTP API 1С (рекомендуется)

В конфигурации 1С добавить метод, например:

- **GET** `/lmb/hs/es/zalogodateli`  
  или  
- **GET** `/lmb/hs/es/contragents?active_zalog=1`

**Ответ:** JSON-массив записей с полями:

- идентификатор (user_uid / id);
- ФИО (first_name, second_name, last_name или full_name);
- телефон (phone);
- email (если есть);
- признак/количество действующих залогов (по желанию).

Логика в 1С: выборка из Справочник.Контрагенты (или аналог), отбор по наличию непогашенного документа Залог (или по регистру). В Laravel после появления метода можно вызывать его через `LmbUserApiService::getZalogodateli()` и отдавать список в приложение/выгрузку.

## Как выдать пользователю lmb права на чтение таблиц

Приложение подключается **к базе lmb** (параметр `dbname=lmb`). Пользователь `lmb` уже может подключаться к этой базе (если выполнено `GRANT ALL PRIVILEGES ON DATABASE lmb TO lmb`).

Важно: в PostgreSQL **права на базу** (ON DATABASE) дают право подключаться и создавать объекты, но **не дают права читать чужие таблицы**. Схема `public` — служебная, по умолчанию у всех есть USAGE (можно обращаться к объектам по имени), но **права на каждую таблицу** задаются отдельно: по умолчанию таблицу может читать только её владелец. Таблицы 1С в `public` принадлежат владельцу (часто `postgres` или пользователю 1С), поэтому для чтения данных пользователю `lmb` нужно выдать **только SELECT на таблицы** (и при необходимости USAGE на схему, если его нет).

Подключаться для выдачи прав нужно под суперпользователем (обычно `postgres`) или владельцем таблиц:

```bash
psql -h 192.168.7.250 -p 5432 -U postgres -d lmb
```

(пароль postgres запрашивается интерактивно; при необходимости укажите `PGPASSWORD=...` в окружении.)

### 1. Право на чтение всех таблиц в схеме public

Схема `public` в PostgreSQL по умолчанию доступна всем (USAGE). Достаточно выдать **только SELECT на таблицы**:

```sql
-- На все существующие таблицы в public (главное — это)
GRANT SELECT ON ALL TABLES IN SCHEMA public TO lmb;

-- Если потребуется работа с последовательностями
GRANT SELECT ON ALL SEQUENCES IN SCHEMA public TO lmb;
```

(Если в вашей установке USAGE на `public` для роли отозван, добавьте: `GRANT USAGE ON SCHEMA public TO lmb;`.)

После этого пользователь `lmb` сможет выполнять `SELECT` по всем таблицам в `public`.

### 2. Право на чтение только конкретных таблиц

Если нужно открыть доступ не всем таблицам, а только, например, справочнику контрагентов и документу залога (имена таблиц нужно взять из метаданных 1С или из `php artisan lmb:db-schema`):

```sql
GRANT USAGE ON SCHEMA public TO lmb;
GRANT SELECT ON public._acc38 TO lmb;           -- пример: одна из таблиц
GRANT SELECT ON public._Reference12345 TO lmb;   -- заменить на реальное имя справочника контрагентов
GRANT SELECT ON public._Document67890 TO lmb;   -- заменить на реальное имя документа залога
```

### 3. Права на таблицы, создаваемые в будущем

Если новые таблицы в `public` создаёт одна и та же роль (например, `postgres` или пользователь 1С), можно задать права по умолчанию, чтобы они автоматически применялись к новым таблицам:

```sql
-- Выполнять от имени владельца создаваемых объектов (часто postgres)
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT ON TABLES TO lmb;
```

Так новые таблицы в `public` тоже будут доступны на чтение для `lmb`.

### 4. Проверка

Под пользователем `lmb`:

```bash
psql -h 192.168.7.250 -p 5432 -U lmb -d lmb -c "SELECT COUNT(*) FROM public._acc38;"
```

Если команда выполняется без ошибки «нет доступа» — права выданы.

### Вариант B: Права на БД + известные имена таблиц

Если нужна выборка именно из PostgreSQL:

1. **Выдать роли `lmb` право SELECT** на таблицы (или представление), в которых хранятся:
   - контрагенты (справочник с ФИО, телефоном, email);
   - документы залога (чтобы отфильтровать «действующих» залогодателей).

2. **Уточнить физические имена таблиц** в БД 1С (через Конфигуратор или выгрузку метаданных в файл, см. `docs/1c_export_metadata_to_file.txt`). Обычно:
   - справочник контрагентов — что-то вроде `public._Reference<ID>`;
   - документ залога — `public._Document<ID>`.

3. **Либо создать в БД представление**, например:
   - `public.v_zalogodateli` (контрагент + ФИО, телефон, email, признак «есть действующий залог»),
   - и выдать `GRANT SELECT ON public.v_zalogodateli TO lmb;`

Тогда в Laravel можно будет делать выборку из этого представления (или из таблиц, если имена занести в конфиг).

- В **LmbUserApiService** добавлен метод **getZalogodateli()**: при появлении в 1С метода вида `GET .../zalogodateli` его можно вызывать из кода.

После появления метода в 1С или выдачи прав на представление/таблицы команду можно доработать под фактический формат ответа или под структуру представления.

## Как ещё можно прочитать данные из таблиц

Пока у пользователя `lmb` нет прав `SELECT` на таблицы, данные из БД напрямую не прочитать. Ниже — все разумные обходные пути.

### 1. Выдать права пользователю lmb (основной способ)

Администратор БД подключается как `postgres` и выполняет:

```sql
GRANT USAGE ON SCHEMA public TO lmb;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO lmb;
```

Дальше читать таблицы можно тем же пользователем `lmb` из приложения. Подробно и скрипт: выше в этом файле и `scripts/grant_lmb_select.sql`.

---

### 2. Подключиться под другим пользователем

Если есть учётная запись с правами на чтение (например, `postgres` или отдельный пользователь для отчётов), можно подключаться к БД под ней:

- В `.env` задать отдельное соединение с другими `LMB_DB_USERNAME` / `LMB_DB_PASSWORD`,  
  **или**
- Добавить в `config/database.php` второе соединение к той же БД с другим пользователем и использовать его только для чтения (команды, отчёты).

Пароли не хранить в репозитории, только в `.env` на сервере.

---

### 3. Представление (VIEW) с правами владельца

Администратор создаёт представление и даёт на него `SELECT` только роли `lmb`:

```sql
-- от имени postgres или владельца таблиц
CREATE VIEW public.v_contragents_read AS
  SELECT _idrref, _code, _description FROM public._acc38;  -- подставить нужную таблицу и поля

GRANT SELECT ON public.v_contragents_read TO lmb;
```

В 1С представление по умолчанию выполняется с правами владельца, поэтому данные из таблицы читаются, а `lmb` видит только результат представления. В приложении делать `SELECT` из `public.v_contragents_read`.

---

### 4. HTTP API 1С

Добавить в конфигурацию 1С метод (например, `GET .../zalogodateli` или `GET .../table/{name}`), который внутри 1С читает нужные справочники/документы и отдаёт JSON. Приложение вызывает этот метод по HTTP (уже есть заготовка в `LmbUserApiService::getZalogodateli()`). К таблицам PostgreSQL приложение не подключается — читает только ответ API.

---

### 5. Выгрузка из 1С в файл

В 1С: отчёт или обработка, которые выгружают нужные данные в файл (JSON, XML, CSV, Excel) на общий каталог или по HTTP. Приложение (Laravel) или cron раз в период забирает файл и импортирует данные. Таблицы БД из приложения по-прежнему не читаются.

---

### 6. Общая сводка

| Способ | Нужны действия | Что читаем |
|--------|----------------|------------|
| GRANT SELECT для lmb | Один раз выполнить SQL на сервере БД | Любые таблицы/представления |
| Другой пользователь БД | Учётная запись с правами + настройка .env/config | Таблицы, к которым есть права у этого пользователя |
| VIEW + GRANT SELECT | Создать VIEW от postgres, выдать SELECT на VIEW роли lmb | Только то, что выбрано во VIEW |
| Метод в HTTP API 1С | Разработка в 1С + вызов из Laravel | То, что возвращает метод |
| Выгрузка в файл из 1С | Обработка/отчёт в 1С + забор файла в Laravel | Содержимое файла |
