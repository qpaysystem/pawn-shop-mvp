# Анализ метаданных 1С (Configuration.xml) и сопоставление с PostgreSQL

Документ описывает структуру выгрузки конфигурации 1С «Ломбард» (Configuration.xml), извлечённые объекты с префиксом **ЛМБ_** и их связь с физическими таблицами в БД 1С (PostgreSQL) для интеграции.

---

## 1. Что такое Configuration.xml

- **Источник:** Конфигуратор 1С → Конфигурация → **Выгрузить конфигурацию в файл**.
- **Формат:** один большой XML (миллионы строк). Кодировка UTF-8.
- **Содержимое:** метаданные конфигурации — типы данных, справочники, документы, регистры, реквизиты, планы обмена и т.д. В файле только **логические имена** (ЛМБ_СкупкаЦенностей, ЛМБ_ВещиВЗалоге); **внутренние числовые ID** платформы (389, 26227), по которым в PostgreSQL строятся имена таблиц `_document389x1`, `_accumrg26227`, в этом формате выгрузки **напрямую не приводятся** (могут быть в виде GUID в тегах Ref/Owner/Тип).

---

## 2. Структура XML

- **Корень:** `<Конфигурация Имя="Ломбард">`.
- **Секции (примеры):**
  - `CatalogObject.Конфигурации` — сведения о конфигурации (имя, версия, дата обновления).
  - `CatalogObject.Объекты` — типы данных (Число, Строка, Дата, СправочникСсылка.*, ДокументСсылка.* и т.д.).
  - `CatalogObject.Свойства` — реквизиты/свойства объектов (Owner ссылается на Объекты по Ref).
  - В других секциях — определения **документов**, **справочников**, **регистров накопления/сведений**: у каждого есть `<Description>`, `<Имя>`, `<Синоним>`, вложенные реквизиты и табличные части.

Поиск по файлу:
- **Документы:** теги с `<Description>ДокументСсылка.ЛМБ_...</Description>` и рядом `<Имя>ЛМБ_...</Имя>`.
- **Регистры накопления:** `<Description>РегистрНакопленияЗапись.ЛМБ_...</Description>`.
- **Справочники:** `<Description>Справочник_ЛМБ_...</Description>` или `СправочникСсылка.ЛМБ_...`.

## 4. Сопоставление с PostgreSQL (таблицы 1С)

Платформа 1С в PostgreSQL создаёт:

| Тип объекта в 1С   | Префикс/шаблон таблицы в PostgreSQL | Пример для ЛМБ |
|--------------------|-------------------------------------|----------------|
| Документ           | `_documentNNN`, расширение реквизитов: `_documentNNNx1` | Скупка: `_document389x1` |
| Табличная часть документа | `_documentNNN_vtMMMM` | Для скупки — своя _vt* (при необходимости смотреть lmb:db-schema) |
| Регистр накопления | `_accumrgNNNN` | ЛМБ_ВещиВЗалоге → одна из `_accumrg*` |
| Справочник         | `_referenceNNN`, расширение: `_referenceNNNx1` | Залоговые билеты/вещи — свои _reference* |

Числа **NNN**, **MMMM**, **NNNN** — внутренние ID метаданных. В Configuration.xml они не заданы в явном виде как числа; связь «логическое имя → физическая таблица» делается одним из способов:

1. **По уже известному соответствию:** документ скупки уже сопоставлен с `_document389x1` по структуре и данным.
2. **По командам приложения:**  
   - `php artisan lmb:db-schema --table=_document389` — список колонок документа;  
   - `php artisan lmb:1c-balances-discovery --count` — список `_accumrg*` и число записей.  
   Сопоставление по количеству записей и по смыслу колонок (контрагент, дата, сумма, склад, номенклатура).
3. **По выгрузке «в файлы»:** Конфигурация → Выгрузить конфигурацию **в файлы** (в папку). В подпапках по типам объектов иногда встречаются внутренние номера или имена, по которым можно точнее определить таблицу.

## 6. Рекомендации для дальнейшей интеграции

1. **Операция по залогу (ЛМБ_ОперацияПоЗалогу)**  
   Найти в PostgreSQL таблицу документа:  
   `php artisan lmb:db-schema --limit=0` и перебор таблиц `_document*` (без _vt), либо фильтр по префиксу `_document` и просмотр колонок. Искать таблицу с полями, похожими на залоговую операцию (контрагент, дата, сумма займа, выкуп и т.д.), и сопоставить по количеству записей с ожидаемым объёмом операций по залогу.

2. **Регистр «Вещи в залоге» (ЛМБ_ВещиВЗалоге)**  
   Выполнить:  
   `php artisan lmb:1c-balances-discovery --count`  
   Выбрать регистр `_accumrg*`, у которого есть `_recordkind` и по смыслу колонок подходят «вещи в залоге» (склад/залог/номенклатура, количество). Задать в .env:  
   `LMB_1C_BALANCES_REGISTER_TABLE=...`, `LMB_1C_BALANCES_DIMENSION_COLUMNS=...`, `LMB_1C_BALANCES_QUANTITY_COLUMN=...`, при необходимости `LMB_1C_BALANCES_AMOUNT_COLUMN=...`.

3. **Справочники ЛМБ_ЗалоговыеБилеты и ЛМБ_ЗалоговыеВещи**  
   При необходимости синхронизации залоговых билетов/вещей — найти таблицы `_reference*` по структуре и количеству записей (lmb:db-schema, lmb:db-tables) и завести конфиг по аналогии с `lmb_1c_contragent_sync`.

4. **Разбор полного Configuration.xml**  
   Для автоматического списка всех ЛМБ_ документов/регистров/справочников:
   ```bash
   php artisan lmb:parse-1c-configuration /Users/evgeny/Documents/Configuration.xml
   ```
   Опции: `--json` — вывод в JSON; без `--lmb-only` по умолчанию выводятся только объекты ЛМБ_*.
