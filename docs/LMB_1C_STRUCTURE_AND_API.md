# Изучение структуры 1С и расширение API LMB

Краткое руководство: как изучить структуру данных базы 1С (testlmb) и как добавить новые методы в веб-сервис 1С, чтобы получать больше данных через API.

---

## 0. Чеклист: что сделать в Конфигураторе для изучения структуры под полноценное API

Если есть доступ в **Конфигуратор 1С**, по шагам ниже вы соберёте всё нужное для проектирования и написания полноценного API. Результат удобно оформить в один текстовый файл (например `docs/1c_structure_notes.md`) и использовать как спецификацию.

### Шаг 1. Составить список основных объектов метаданных

1. **Конфигурация** → **Открыть конфигурацию**.
2. В дереве **Метаданные** пройдите по разделам и выпишите **имена** объектов, которые относятся к ломбарду/контрагентам/залогам/товарам:
   - **Справочники** — например: `Контрагенты`, `Номенклатура`, `Склады`, `ВидыЗалогов` и т.д.
   - **Документы** — например: `Залог`, `Выкуп`, `Реализация`, `Приход` и т.д.
   - **Регистры сведений** — настройки, классификаторы.
   - **Регистры накопления** — остатки, движения (например остатки по складам, по залогам).
   - **Регистры бухгалтерии** — если используется учёт.

Для каждого объекта записывайте: **синоним (имя на русском)**, **реквизиты** (имя, тип), **табличные части** (для документов/справочников). Это можно копировать из свойств объекта в Конфигураторе.

### Шаг 2. Выписать ключевые реквизиты для API

По каждому объекту, который планируете отдавать через API, зафиксируйте:

- **Уникальный идентификатор** (обычно `Ссылка` или код; для справочников часто есть реквизит `Код`, `УникальныйИдентификатор` или аналог).
- **Поля для выдачи в JSON**: наименование, дата, контрагент, сумма, склад, номенклатура и т.д.
- **Связи**: какой реквизит ссылается на контрагента, на номенклатуру, на склад — чтобы строить отборы и JOIN в запросах.

Пример записи:

```text
Справочник.Контрагенты
  - Ссылка (уникальный ключ)
  - Наименование, Код
  - ИНН, Телефон, Email (если есть)
  - Реквизиты для поиска по телефону: Телефон или контактная информация
```

### Шаг 3. Проверить запросами в Консоли запросов

1. **Сервис** → **Консоль запросов** (или **Отладка** → **Консоль запросов**).
2. Для каждого важного объекта выполните тестовый запрос и зафиксируйте **реальные имена полей** и формат данных:

Примеры:

```1c
// Контрагенты (первые 3)
ВЫБРАТЬ ПЕРВЫЕ 3
    К.Ссылка,
    К.Наименование,
    К.Код,
    К.Телефон
ИЗ Справочник.Контрагенты К

// Документы залога (структура)
ВЫБРАТЬ ПЕРВЫЕ 2
    З.Ссылка,
    З.Номер,
    З.Дата,
    З.Контрагент,
    З.Сумма
ИЗ Документ.Залог З
```

Имена объектов (`Контрагенты`, `Залог`) замените на те, что есть в вашей конфигурации. Результат запроса покажет точные имена колонок и типы — их и использовать в коде API.

### Шаг 4. Найти текущий HTTP-сервис и его методы

1. В метаданных: **Общие** → **HTTP-сервисы** (или **Веб-сервисы**).
2. Найдите сервис, соответствующий публикации `/lmb/hs/es` (часто имя типа `es` или по пути).
3. Откройте **Методы** этого сервиса. Выпишите:
   - имена методов (`user`, `ostatki`, `user_no` и т.д.);
   - какой HTTP-метод используется (GET/POST);
   - какие параметры принимает метод (URL, тело запроса).

По возможности откройте **модуль** метода и посмотрите, откуда берутся данные (какой запрос, какой справочник/документ/регистр). Это даст понимание, как устроен текущий API и по какому образцу делать новые методы.

### Шаг 5. Сформировать список методов для полноценного API

На основе списка объектов и реквизитов составьте таблицу желаемых методов. Пример:

| Метод API | Назначение | Источник данных в 1С | Параметры |
|-----------|------------|----------------------|-----------|
| GET /user/{phone} | Контрагент по телефону | Справочник.Контрагенты, отбор по телефону | phone |
| GET /ostatki | Остатки/каталог | Регистр или документы | — |
| GET /contragent/{id} | Контрагент по ID | Справочник.Контрагенты по Ссылке | id |
| GET /zalogi?contragent={id} | Залоги контрагента | Документ.Залог, отбор по Контрагент | contragent |
| GET /nomenclature | Номенклатура | Справочник.Номенклатура | — |

Имена объектов и полей подставьте из шагов 1–3. Этот список станет планом разработки: по одному методу в 1С + вызов из Laravel.

### Шаг 6. (По желанию) Выгрузить конфигурацию в файлы

Если нужно хранить описание структуры в репозитории или показывать её в Cursor:

- **Конфигурация** → **Выгрузить конфигурацию в файлы** — выберите каталог (например `docs/1c_metadata_export`). Получите набор XML/текстовых файлов с метаданными.
- Этот каталог можно добавить в проект и при необходимости анализировать в редакторе (поиск по именам справочников, реквизитов).

### Шаг 7. Одна процедура 1С: вся структура метаданных в один файл

В языке **запросов** 1С (ВЫБРАТЬ … ИЗ) нельзя получить метаданные — только данные из таблиц. Чтобы выгрузить **всю структуру таблиц и метаданных в один файл**, нужен код на **встроенном языке** 1С, который обходит объект `Метаданные` и пишет результат в файл.

В проекте лежит готовый код: **[docs/1c_export_metadata_to_file.txt](1c_export_metadata_to_file.txt)**. В нём процедура `ВыгрузитьМетаданныеВФайл(ПутьКФайлу)`:

- обходит Справочники, Документы, Регистры сведений/накопления/бухгалтерии;
- для каждого объекта выводит имя, синоним, реквизиты (и типы), табличные части, измерения и ресурсы;
- записывает всё в один текстовый файл в UTF-8.

**Как использовать:** создать обработку в Конфигураторе, вставить код из файла в модуль обработки, вызвать процедуру с путём к файлу (например в каталог `docs/` проекта). Полученный файл можно открыть в Cursor и использовать как спецификацию для проектирования API.

После выполнения чеклиста у вас будет: список объектов, реквизиты, примеры запросов, текущие методы HTTP-сервиса и план новых методов API. Дальше — реализация методов в Конфигураторе (раздел 2 ниже) и вызовы из Laravel.

## 1. Как изучить структуру таблиц/данных 1С

Структуру можно смотреть с двух сторон: **логическая** (метаданные 1С) и **физическая** (таблицы в СУБД).

### 1.1. Логическая структура — метаданные в Конфигураторе 1С

Это основной способ без доступа к PostgreSQL/SQL Server.

1. Запустите **1С:Предприятие** в режиме **Конфигуратор**.
2. Подключитесь к базе **testlmb** (опубликованная база или по имени сервера).
3. Откройте **Конфигурация** → **Открыть конфигурацию**.
4. В дереве **Метаданные** будут:
   - **Справочники** — контрагенты, номенклатура, склады и т.д. (реквизиты, табличные части).
   - **Документы** — приход, реализация, залоги и т.д.
   - **Регистры сведений / накопления / бухгалтерии** — остатки, движения, обороты.

По каждому объекту видно: **реквизиты** (поля), **табличные части**, **связи**. Это и есть «структура» с точки зрения 1С. Имена физических таблиц в БД (типа `_Reference123`) здесь не показываются — они генерируются платформой.

**Консоль запросов:** в Конфигураторе — **Сервис** → **Консоль запросов** (или **Отладка** → **Консоль запросов**). Там можно писать запросы на языке 1С к этим объектам (например, `ВЫБРАТЬ * ИЗ Справочник.Контрагенты`) и смотреть результат. Так вы поймёте, какие поля и таблицы доступны для выгрузки в API.

### 1.2. Физическая структура — таблицы в СУБД

Если есть **логин и пароль к базе данных** (их задаёт администратор в консоли управления сервером 1С):

- В проекте уже есть:
  - подключение к БД 1С (PostgreSQL: `LMB_DB_DRIVER=pgsql`, `LMB_DB_HOST`, `LMB_DB_PORT`, и т.д. в `.env`);
  - команда просмотра структуры:  
    `php artisan lmb:db-schema`  
  Она выводит список таблиц и колонок (физические имена в СУБД).
  - команда списка таблиц по префиксу:  
    `php artisan lmb:db-tables --prefix=_reference --no-vt` (см. `docs/LMB_1C_TABLES_STRUCTURE_AND_SYNC.md`).
- Скрипт поиска хоста с открытым портом СУБД в VPN:  
  `./scripts/find_mssql_host_vpn.sh`

**Подробно:** имена таблиц 1С, связи между ними, уже определённые справочники (контрагенты-физлица, банки) и план синхронизации описаны в **`docs/LMB_1C_TABLES_STRUCTURE_AND_SYNC.md`**.

Без учётных данных к БД физические таблицы из приложения посмотреть нельзя — только через Конфигуратор (метаданные).

### 1.3. Что уточнить у администратора 1С

- Логин и пароль к СУБД для базы testlmb (если нужен просмотр физических таблиц).
- Список опубликованных HTTP-сервисов и их методы (какие URL уже есть кроме `/lmb/hs/es`).
- Какие справочники/документы/регистры считаются основными для интеграции (контрагенты, залоги, номенклатура и т.д.).

## 2. Как написать более расширенное API в 1С

Текущий API — это **HTTP-сервис 1С** по адресу `http://5.128.186.3/lmb/hs/es`. Методы (например, `user`, `ostatki`, `user_no`) реализованы в конфигурации 1С. Чтобы расширить API:

### 2.1. Где в 1С искать веб-сервис

1. В **Конфигураторе** откройте конфигурацию базы testlmb.
2. В дереве метаданных: **Общие** → **HTTP-сервисы** (или **Веб-сервисы** / **REST** — в зависимости от версии и конфигурации).
3. Найдите сервис, который публикуется по пути `/lmb/hs/es` (имя может быть `es` или другим). В нём — **методы** (например, `user`, `ostatki`, `user_no`).

### 2.2. Добавить новый метод в существующий HTTP-сервис

1. Откройте нужный HTTP-сервис → **Методы**.
2. Создайте **новый метод** (имя станет частью URL, например `catalog` → вызов будет `GET .../es/catalog` или как настроено).
3. В обработчике метода напишите код на встроенном языке 1С:
   - выполните запрос к метаданным (справочники, документы, регистры);
   - сформируйте JSON (через `ЗаписатьJSON` или построение строки);
   - верните ответ с типом `application/json`.

Пример идей для новых методов:

- **Справочник номенклатуры:** выборка из `Справочник.Номенклатура` (код, наименование, тип и т.д.) → JSON.
- **Залоги по контрагенту:** запрос к документу залога или регистру с отбором по контрагенту → список залогов с суммами, датами.
- **Остатки по складу/номенклатуре:** запрос к регистру остатков → JSON.

Точные имена объектов и полей нужно смотреть в метаданных (п. 1.1).

### 2.3. Публикация и авторизация

- После изменений конфигурации её нужно **обновить на сервере** и убедиться, что **HTTP-сервис опубликован** (в настройках публикации базы на веб-сервере путь `/lmb/hs` или `/lmb/hs/es` должен указывать на эту базу и сервис).
- Текущая интеграция использует **Basic Auth** (UserWebServis). Новые методы обычно делают с той же авторизацией, чтобы наш `LmbUserApiService` мог вызывать их без изменений (достаточно добавить вызовы новых URL).

### 2.4. Вызов нового API из нашего приложения (Laravel)

После появления в 1С нового метода (например, `GET .../es/catalog`):

1. В `App\Services\LmbUserApiService` добавить метод, например:
   - `getCatalog(): ?array` — запрос `GET $this->baseUrl . '/catalog'`, разбор JSON.
2. Использовать этот метод в контроллерах, командах или задачах по необходимости.

Формат ответа (поля, вложенность) лучше зафиксировать в 1С и обрабатывать в Laravel по контракту (как сейчас с `user_uid`, `first_name` и т.д. для контрагента).

## 3. Краткая схема

| Цель | Действие |
|------|-----------|
| Понять, какие сущности и поля есть в 1С | Конфигуратор → Метаданные + Консоль запросов |
| Увидеть физические таблицы СУБД | Получить учётные данные БД → `php artisan lmb:db-schema` |
| Добавить новый метод API | Конфигуратор → HTTP-сервис → новый метод → код запроса к метаданным и вывод JSON |
| Вызвать новый метод из Laravel | Новый метод в `LmbUserApiService` + использование в коде |

Если опишете, какие именно данные нужны (например, «залоги по клиенту», «номенклатура», «остатки по складу»), можно расписать пример кода метода в 1С и вызов из Laravel под ваш кейс.
