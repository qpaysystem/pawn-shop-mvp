# Обзор проекта «Ломбард MVP» (pawn-shop-mvp)

Документ для онбординга и планирования: что это за система, как устроена, где «сырые» места и куда двигаться дальше.

---

## 1. Назначение

Веб-приложение для **сети ломбардов и комиссионных магазинов**: учёт клиентов и товаров, договоры **залога** и **комиссии**, **скупка**, касса и расширенный бухгалтерский контур, **публичный лендинг**, маркетинг и колл-центр. Параллельно развивается **чтение данных из БД 1С** (выгрузка PostgreSQL конфигурации «Ломбард») для синхронизации клиентов, договоров, остатков и справочной информации — без замены 1С как учётной системы.

---

## 2. Стек и окружение

| Компонент | Технология |
|-----------|------------|
| Backend | PHP 8.1+, **Laravel 10** |
| Основная БД приложения | **MySQL 8** (или PostgreSQL по `DB_*`) |
| БД 1С (только чтение) | **PostgreSQL** — отдельное подключение `lmb_1c_pgsql`, включается при `LMB_DB_DRIVER=pgsql` |
| Frontend | **Blade**, Bootstrap |
| Зависимости | Composer; для таблиц — PhpSpreadsheet и др. |

Локально: `php artisan serve`, миграции и сиды — см. [README.md](../README.md). Боевой деплой — [DEPLOY.md](../DEPLOY.md).

## 3. Как устроено приложение

### 3.1. Роли и доступ

- **super-admin** — все магазины, пользователи, настройки.
- **manager** — полный доступ в рамках своего `store_id`.
- **appraiser** — приём товара (залог/комиссия).
- **cashier** — продажи и выкупы.
- **storekeeper** — статусы товара и места хранения.

Пользователь (кроме супер-админа) привязан к одному магазину.

### 3.2. Основной операционный контур

1. **Приём товара** — клиент, тип договора (залог/комиссия), карточка товара, фото, суммы, печать.
2. **Товары** — список, фильтры, история статусов, связь с договором.
3. **Договоры залога / комиссии** — списки, просмотр, выкуп/продажа.
4. **Скупка** — отдельные сущности `purchase_contracts` и связанные `items`.

Справочники: категории, бренды, статусы, места хранения, магазины.

### 3.3. Расширенные модули (уже в коде)

- **Касса**, банковские счета и выписки, статьи расходов.
- **План счетов / проводки** (шаблоны документов, ledger).
- **Зарплата** (начисления).
- **Маркетинг** (в т.ч. импорт статистики 2ГИС), **колл-центр**, **база знаний**.
- **Лендинг** ломбарда (публичные страницы, каталог и т.д.) — см. маршруты в `routes/web.php`.

Точная степень «боевой готовности» модулей разная: часть сделана под реальные процессы, часть — задел под развитие.

### 3.4. Интеграция с 1С (LMB)

Данные **не пушатся обратно в 1С** из этого приложения в рамках описанных команд — идёт **импорт/синхронизация в нашу БД**.

| Область | Пример команд | Конфиг / документация |
|--------|----------------|------------------------|
| Клиенты (контрагенты) | `lmb:sync-contragents`, опции backfill паспортов и identity | `config/services.php` → `lmb_1c_contragent_sync`, [LMB_SYNC_CLIENTS.md](LMB_SYNC_CLIENTS.md) |
| Паспорта из регистра | `lmb:sync-passports-from-1c` | Правило Cursor `.cursor/rules/lmb-1c-passport-sync.mdc` |
| Договоры залога | `lmb:sync-pawn-contracts` | `lmb_1c_pawn_sync`, [LMB_SYNC_PAWN.md](LMB_SYNC_PAWN.md) |
| Скупка | `lmb:sync-purchase-contracts` | `lmb_1c_purchase_sync` и др. |
| Филиалы | `lmb:sync-stores-from-1c` | `lmb_1c_stores_sync` |
| Остатки / балансы | `lmb:sync-1c-balances` и см. доки | [LMB_1C_BALANCES_SYNC.md](LMB_1C_BALANCES_SYNC.md) |
| Разведка схемы 1С | `lmb:db-schema`, `lmb:discover-1c-documents`, и т.д. | папка `docs/LMB_*.md` |

Полная картина сценариев: [LMB_INTEGRATION_FULL.md](LMB_INTEGRATION_FULL.md).

Задачи VS Code/Cursor для типовых команд: `.vscode/tasks.json`.

## 4. Что условно «не доделано» или требует внимания

- **Двусторонняя синхронизация с 1С** — по сути нет: приложение опирается на чтение PostgreSQL; обратная запись в 1С не описана как единый контур.
- **Имена таблиц/полей 1С** зависят от версии конфигурации (`_documentNNN`, `_fldXXXX`). После обновления 1С возможны сдвиги — нужны проверки через `lmb:db-schema` и discover-команды.
- **Память PHP при полном синке контрагентов** — при десятках тысяч записей включены ограничения preload (адрес по ФИО, тексты док. 517) и повышение `memory_limit` в команде; при необходимости — дозаполнение батчами (`--backfill-identity-from-1c`). См. [LMB_SYNC_CLIENTS.md](LMB_SYNC_CLIENTS.md).
- **Модели Laravel 10** — касты должны задаваться свойством **`$casts`**, а не методом `casts()` (как в Laravel 11). В `Client` это исправлено; в других моделях стоит проверить и привести к одному стилю, иначе касты фактически не работают.
- **Автотесты** — каталог `tests/` может отсутствовать или быть минимальным; регрессии по синку и формам пока ловятся вручную.
- **Секреты и доступ к БД 1С** — только в `.env`, не коммитить; VPN/SSH — см. [LMB_SSH_AND_DB_ACCESS.md](LMB_SSH_AND_DB_ACCESS.md).

## 5. На что обратить внимание при эксплуатации

1. **Порядок синка**: сначала клиенты, затем договоры, которые ссылаются на `client_id` (см. LMB_INTEGRATION_FULL).
2. **Миграции** после обновления кода: `php artisan migrate` (в т.ч. поля LMB у `clients`, `items`, договоров).
3. **Резервное копирование** своей БД приложения отдельно от политики бэкапов 1С.
4. **Хостинг**: Document Root на `public/`, `storage:link`, права на `storage/` и `bootstrap/cache/`.
5. **Производительность синка**: большие выборки из 1С лучше запускать вне пиковых часов; при OOM — уменьшить объём одной команды или увеличить `memory_limit` осознанно.

## 6. Рекомендуемые дальнейшие шаги

1. **Зафиксировать целевой сценарий**: только зеркало для отчётов/CRM или полноценная замена части операций вне 1С — от этого зависит приоритет доработок.
2. **Привести модели к `$casts`** там, где ещё остался метод `casts()` под Laravel 11.
3. **Добавить smoke-тесты** (хотя бы PHPUnit для критичных сервисов синка с моком БД).
4. **Документировать «эталонные» имена таблиц 1С** для вашей боевой базы (одна страница в `docs/` по факту `discover`).
5. **Планировщик (cron)** для регулярного синка: контрагенты → залоги/скупка → балансы (по необходимости), с логированием и алертами при ошибках.
6. **UI/UX**: единые экраны статуса последнего синка и ошибок для администратора.
7. **Опционально**: вынос тяжёлого синка в очередь Laravel (jobs) вместо одной длинной CLI-команды.
