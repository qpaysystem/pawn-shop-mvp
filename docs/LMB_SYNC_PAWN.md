# Синхронизация действующих залогов из 1С

Залоги (документ «Залог» в 1С) синхронизируются в таблицу `pawn_contracts`: клиент, номенклатура (товар), сумма займа, процент, даты, сумма выкупа.

## Важно: _document389x1 — это скупка, не залог

В базе lmb документ **\_document389x1** (168 записей) — это **договоры скупки** (покупка товара у клиента), а не залоги. Если вы ошибочно загрузили их через `lmb:sync-pawn-contracts`, перенесите их в раздел «Договоры скупки»:

```bash
php artisan lmb:move-pawn-from-1c-to-purchase --dry-run   # посмотреть количество
php artisan lmb:move-pawn-from-1c-to-purchase            # перенести
```

Таблицу документа **залога** (займ под залог) нужно определить по метаданным 1С отдельно; в .env `LMB_1C_PAWN_DOCUMENT_TABLE` оставьте пустым, пока не уточните.

Номенклатура: при синхронизации из 1С товар (item) создаётся по табличной части или с именем «Залог 1С {номер}». Если в 1С по документу нет привязки к номенклатуре или колонка в конфиге неверна — привязка к товару будет условной (можно отредактировать товар вручную или настроить колонку номенклатуры в .env).

### Синхронизация договоров скупки и товаров из 1С

Команда создаёт или обновляет договоры скупки (`purchase_contracts`) и товары (`items`) по документам из таблицы скупки 1С (`_document389x1`):

```bash
php artisan lmb:sync-purchase-contracts --dry-run   # только подсчёт
php artisan lmb:sync-purchase-contracts             # запись в БД
```

- По каждому документу 1С: ищется клиент по контрагенту (`user_uid`), при отсутствии клиента запись пропускается.
- Если договор с таким `lmb_doc_uid` уже есть — обновляются сумма, дата, клиент, магазин; при отсутствии товара создаётся новый.
- Если договора нет — создаётся товар (наименование из колонок _fld9638/_fld9643/_fld9650 или «Скупка 1С {номер}») и договор скупки.

Конфиг: `config/services.php` → `lmb_1c_purchase_sync` (LMB_1C_PURCHASE_* в .env).

### Наименования товаров договоров скупки из заголовка документа 1С

У документа скупки (_document389x1) табличные части пусты; наименование товара можно взять из заголовка документа (текстовые поля _fld9638, _fld9643, _fld9650 — по порядку, первая непустая записывается в item.name):

```bash
php artisan lmb:refresh-purchase-item-names-from-1c
```

Команда также пытается **разобрать описание** (если оно в формате вида «Цепь, Золото 585, 8,67 гр.») и заполнить в `items` поля:\n- `metal` (Золото/Серебро/…)\n- `sample` (585/750/…)\n- `weight_grams` (8.670)\n+
### Если в БД 1С нет описания товара (импорт из отчёта 1С)

В некоторых конфигурациях/периодах в таблице документа скупки `_document389x1` текстовые поля с описанием товара могут быть пустыми. Тогда описание можно подтянуть из **отчёта 1С** (TSV/таблица, как в вашей выгрузке) и обновить товары у договоров скупки по номеру `00БП-******`:

```bash
php artisan lmb:import-purchase-items-from-1c-report /path/to/report.tsv --dry-run
php artisan lmb:import-purchase-items-from-1c-report /path/to/report.tsv
```

При нескольких строках в одном документе 1С: в `item.description` сохраняется список всех позиций, а в `item.name` — первая позиция (с пометкой «и ещё N»).

Конфиг: `config/services.php` → `lmb_1c_purchase_item_name`. В .env при необходимости задайте:
- `LMB_1C_PURCHASE_DOCUMENT_TABLE=_document389x1`
- `LMB_1C_PURCHASE_NAME_COLUMNS=_fld9638,_fld9643,_fld9650`

## Чек-лист для базы 192.168.7.250 / lmb

1. **Клиенты из 1С уже в приложении** — иначе залоги без совпадения контрагента пропускаются:
   ```bash
   php artisan lmb:sync-contragents
   ```
2. **В .env заданы** (уже прописано для этой базы):
   - `LMB_DB_*` — хост 192.168.7.250, база lmb, пользователь lmb
   - `LMB_1C_PAWN_DOCUMENT_TABLE=_document389x1` — таблица с залогами (168 записей)
   - колонки контрагента, даты, номера, суммы, процента, окончания, табличная часть с номенклатурой
3. **Запуск синхронизации:**
   ```bash
   php artisan lmb:sync-pawn-contracts --dry-run   # проверка
   php artisan lmb:sync-pawn-contracts             # только действующие
   php artisan lmb:sync-pawn-contracts --all       # все залоги
   ```

## Что нужно перед запуском (для другой базы)

1. **Определить таблицу документа «Залог»** в БД 1С (например, через выгрузку метаданных или перебор `php artisan lmb:db-tables --prefix=_document --no-vt --count`). Типичное имя: `_document31784` или похожее.
2. **Узнать имена колонок** документа (контрагент, дата, номер, сумма займа, процент, дата окончания, сумма выкупа). Команда: `php artisan lmb:db-schema --table=_document31784`.
3. При необходимости — **табличная часть** (строки с номенклатурой и суммами): `_document31784_vt31892` и колонки: ссылка на документ, номенклатура (rref), сумма, наименование.

## Запуск

```bash
# Проверка конфига и подключения (без записи)
php artisan lmb:sync-pawn-contracts --dry-run

# Синхронизация только действующих залогов
php artisan lmb:sync-pawn-contracts

# Синхронизация всех залогов (включая просроченные)
php artisan lmb:sync-pawn-contracts --all
```

После синхронизации договоры отображаются в разделе «Документы» → «Договоры залога» и в карточке клиента.

## Поведение

- По каждому документу залога (или по каждой строке табличной части, если она задана) создаётся или обновляется запись в `pawn_contracts`.
- Клиент ищется по `user_uid` = hex ссылки на контрагента 1С. Если клиент не найден (не синхронизирован из 1С), запись пропускается.
- Товар (номенклатура): если задана табличная часть с номенклатурой, ищется `Item` с `lmb_ref` = hex ссылки; иначе создаётся новый товар с наименованием из строки или «Залог 1С {номер}».
- Уникальность по 1С: поле `lmb_doc_uid` в `pawn_contracts` (ID документа в hex или doc_uid + номер строки при синхронизации по ТЧ).
- Действующие залоги: при `LMB_1C_PAWN_ONLY_ACTING=true` и заданной `LMB_1C_PAWN_EXPIRY_COLUMN` выбираются только документы с датой окончания ≥ текущей даты.
