# Структура таблиц 1С LMB: связи, смысл данных, синхронизация

Документ описывает **физическую структуру БД 1С** (PostgreSQL), типы таблиц, связи между ними и уже используемые таблицы для синхронизации с приложением ломбарда.

**Подключение:** хост 192.168.7.250, база `lmb`, пользователь `lmb`. См. `config/database.php` (соединение `lmb_1c_pgsql`) и `docs/LMB_SSH_AND_DB_ACCESS.md`.

**Продолжать анализ структуры БД:** методика и команды — **docs/LMB_1C_POSTGRES_ANALYSIS.md**; быстрый обзор категорий таблиц и крупнейших таблиц: `php artisan lmb:postgres-structure-overview`.

---

## 1. Именование таблиц 1С в PostgreSQL

Платформа 1С хранит объекты в таблицах с **внутренними именами** (по ID метаданных). Соответствие «логическое имя в Конфигураторе → физическая таблица» даёт только выгрузка метаданных из 1С (см. раздел 6). Логические имена объектов конфигурации «Ломбард» (Контрагенты, ЛМБ_ОперацияПоЗалогу, РС_ИсточникРекламы и др.) описаны в **docs/LMB_1C_METADATA_LOMBARD.md**.

| Префикс / шаблон | Назначение |
|------------------|------------|
| `_referenceNNN` | **Справочники** (каталоги). Основная таблица элемента справочника. |
| `_referenceNNNx1` | То же справочник, **расширенная** таблица (доп. реквизиты, подчинённые данные). Часто именно здесь лежат реквизиты вроде телефона, паспорта. |
| `_referenceNNN_vtMMMM` | **Табличная часть** справочника (подчинённая таблица). |
| `_documentNNN` | **Документы**. Заголовок документа (дата, номер, контрагент, сумма и т.д.). |
| `_documentNNN_vtMMMM` | Табличная часть документа (строки). |
| `_acc38`, `_accrg*`, `_accrgat*` | План счетов и **регистры бухгалтерии** (движения, обороты). |
| `_inforeg*` | **Регистры сведений**. |
| `_accumreg*` | **Регистры накопления** (остатки, обороты). |

Числа `NNN`, `MMMM` — внутренние ID объектов в метаданных 1С. Без выгрузки метаданных или Конфигуратора однозначно сопоставить ID с именем (например, «Контрагенты», «Залог») нельзя; можно только по количеству записей и по названиям полей угадывать назначение.

## 2. Общие колонки в таблицах 1С

Почти во всех таблицах справочников и документов есть:

| Колонка | Тип | Смысл |
|---------|-----|--------|
| `_idrref` | bytea | Уникальный идентификатор записи (ссылка 1С). В запросах удобно: `encode(_idrref, 'hex')` для строкового ключа. |
| `_marked` | boolean | Пометка на удаление. В выборках обычно: `WHERE NOT _marked`. |
| `_version` | integer | Версия записи (оптимистичная блокировка). |
| `_description` | mvarchar | Часто «наименование» (для справочников). |
| `_code` | mchar/mvarchar | Код элемента (не всегда заполнен). |
| `_parentidrref` | bytea | Ссылка на родителя (для иерархических справочников). |
| `_folder` | boolean | Признак «это группа/папка», а не элемент. |
| `_fldNNNN` | разные | Реквизиты объекта. Номер поля (NNNN) — внутренний ID реквизита в метаданных. |

Ссылки на другие объекты хранятся в колонках типа `*rref` (bytea). Например, документ может иметь `_fld3195rref` → ссылка на справочник контрагентов. Связь: значение в `_fld3195rref` совпадает с `_idrref` в таблице контрагентов.

## 3. Уже определённые таблицы (для синхронизации)

### 3.1. Справочник контрагентов — физлица (залогодатели)

**Полная сборка полей залогодателя из PostgreSQL** (ФИО, телефон, паспорт, вид документа, прописка, vt3220): см. **docs/LMB_1C_CONTRAGENT_FULL_DATA.md**.

| Параметр | Значение |
|----------|----------|
| Таблица | `public._reference122x1` |
| Записей (без пометки) | ~23 650 |
| Назначение | Физические лица: ФИО, телефон, паспорт. Используется для синхронизации в раздел «Клиенты». |

**Ключевые колонки:**

| Колонка | Тип | Смысл |
|---------|-----|--------|
| `_idrref` | bytea | Уникальный ID (в приложении сохраняем как `user_uid` в hex). |
| `_description` | mvarchar(100) | Краткое наименование. |
| `_fld3178` | mvarchar(250) | **ФИО** (полное). |
| `_fld41084` | mvarchar(15) | **Телефон**. |
| `_fld3184` | mvarchar(9) | Серия паспорта (в части баз заполнено у немногих). |
| `_fld3185` | mvarchar(10) | Номер паспорта. |
| `_fld3201` | mvarchar(70) | **Номер паспорта** (в текущей базе заполнен у ~23,5 тыс.). |
| `_fld3202` | mvarchar(70) | **Серия паспорта** (в текущей базе заполнен у ~23,5 тыс.). |

В конфиге `lmb_1c_contragent_sync` по умолчанию используются **passport_series_column** = `_fld3202`, **passport_number_column** = `_fld3201` (если в вашей базе паспорт в других полях — задайте в .env `LMB_1C_CONTRAGENT_PASSPORT_SERIES_COLUMN` и `LMB_1C_CONTRAGENT_PASSPORT_NUMBER_COLUMN`).
| Остальные `_fld*` | — | Адрес, даты, контакты и т.д. |

Конфиг: `config/services.php` → `lmb_1c_contragent_sync` (table, phone_column, name_column, passport_series_column, passport_number_column). Синхронизация: `php artisan lmb:sync-contragents`.

**Как подгрузить паспортные данные из 1С массово**

1. **Полная синхронизация контрагентов** (рекомендуется один раз прогнать):
   ```bash
   php artisan lmb:sync-contragents
   ```
   Команда читает из 1С всех контрагентов (физлица), находит нашего клиента по **user_uid**, по точному **телефону** или по **phone_key** (последние 10 цифр номера), обновляет ФИО, **паспортные данные**, дату создания в 1С, источник рекламы. Для сопоставления по номеру телефона в таблице `clients` используется колонка `phone_key` (заполняется миграцией и при сохранении клиента).

2. **Только паспорта** (без перезаписи остальных полей):
   ```bash
   php artisan lmb:sync-passport-from-1c
   ```
   Подтягивает паспорт из 1С только для уже существующих клиентов (по user_uid или по последним 10 цифрам телефона). Опции: `--stats` — только статистика, `--dry-run` — без записи в БД.

3. **Требования:** в `.env` заданы `LMB_DB_DRIVER=pgsql`, хост/порт/база 1С. У клиентов в нашей базе в поле «Телефон» должно быть не менее 10 цифр (любой формат: 8 953…, +7…, 7953…), чтобы сработало сопоставление по `phone_key`.

### 3.2. Справочник филиалов (подразделения)

| Параметр | Значение |
|----------|----------|
| Таблица | `public._reference197` |
| Записей | 65 |
| Назначение | Филиалы/точки: «Дуси Ковальчук 266/2», «Горский, 1», «Вокзал ювелирка» и т.д. Синхронизация в `stores`: `php artisan lmb:sync-stores-from-1c`. Конфиг: `lmb_1c_stores_sync` в config/services.php. |

**Только склады из документов (хранение товара).** В справочнике _reference197 — 65 записей (в т.ч. служебные подразделения). Склады, реально используемые для хранения в документах залога и скупки, берутся из колонок документа: залог — `LMB_1C_PAWN_STORE_COLUMN` (напр. `_fld41704rref` для _document41694x1), скупка — `LMB_1C_PURCHASE_STORE_COLUMN` (по умолчанию `_fld9627rref`). Команда с опциями:

- `php artisan lmb:sync-stores-from-1c --only-used` — создать/обновить в `stores` только те склады, которые встречаются в этих документах.
- `php artisan lmb:sync-stores-from-1c --only-used --clear` — перепривязать все зависимости (договоры, визиты, касса и т.д.) к первому такому складу, удалить остальные магазины и создать только «используемые». Если справочник _reference197 не содержит этих ссылок (склад из другого справочника 1С), записи создаются с условными именами «Склад XXXXXXXX».

### 3.3. Справочник контрагентов — юр. лица / банки

| Параметр | Значение |
|----------|----------|
| Таблица | `public._reference108` |
| Записей | ~1 390 |
| Назначение | Организации, банки. **Не** используется для клиентов приложения (мы синхронизируем только физлиц из `_reference122x1`). |

**Ключевые колонки:** `_description`, `_code`, `_fld2988` (телефон/код), `_fld2989` (ИНН) и др. Для синхронизации клиентов ломбарда эта таблица не нужна.

### 3.4. Связь справочников 108 и 122

- `_reference122` — основная таблица справочника (у нас 0 записей).
- `_reference122x1` — расширенная таблица того же справочника, в ней лежат все элементы с реквизитами (ФИО, телефон, паспорт). В синхронизации используем только `_reference122x1`.

## 4. Связи между таблицами

- **Ссылки:** колонки с суффиксом `rref` (bytea) хранят ссылку на другую таблицу: значение равно `_idrref` соответствующей записи.
- **Сравнение:** в SQL можно сравнивать два `bytea` или использовать `encode(_idrref, 'hex')` для вывода и сопоставления с нашим `user_uid`.
- **Документ → контрагент:** в документе (залог, выкуп и т.д.) обычно есть поле вида `_fldXXXXrref` — ссылка на справочник контрагентов. Чтобы получить залоги по клиенту, нужно выбрать документы, у которых эта ссылка равна `_idrref` выбранного контрагента из `_reference122x1`.

Табличные части документов (`_documentNNN_vtMMMM`) связаны с заголовком по полю типа `_documentNNN_idrref` = `_idrref` заголовка.

## 5. Документы (залоги, выкупы и т.д.)

Документы в БД — таблицы `_documentNNN`. Табличные части — `_documentNNN_vtMMMM`.

- Чтобы **найти документ «Залог»** без метаданных 1С: перебрать таблицы `_document*` (без `_vt`), посмотреть структуру и количество записей; искать таблицы с полями типа `_date`/`_datetime`, полем-ссылкой на контрагента и числовыми полями (сумма).
- В 1С один документ может иметь несколько табличных частей (товары, суммы по строкам и т.д.) — каждая в своей `_vt*` таблице.

Для **синхронизации залогов/выкупов** с нашей БД в будущем понадобится:
1. Определить точные имена таблиц документа «Залог» (и при необходимости «Выкуп», «Реализация»).
2. Выписать колонки: дата, номер, контрагент (rref на _reference122x1), сумма, склад, номенклатура (если есть).
3. Связывать по `encode(_fldXXXrref, 'hex')` с `clients.user_uid` (если user_uid у нас хранится в hex из _reference122x1._idrref).

## 6. Как узнать соответствие «таблица ↔ объект метаданных 1С»

Без доступа к Конфигуратору 1С однозначно сопоставить `_reference122` с именем справочника нельзя. Рекомендуется:

1. **Выгрузка метаданных в файл** (см. `docs/1c_export_metadata_to_file.txt`): в Конфигураторе запустить обработку — в файл попадёт список справочников, документов, регистров и их реквизиты (логические имена). По номерам полей и количеству записей можно сопоставить файл с физическими таблицами в PostgreSQL.
2. **Консоль запросов 1С:** в Конфигураторе выполнять запросы к объектам метаданных (например, `ВЫБРАТЬ * ИЗ Справочник.Контрагенты`) — станет ясно, какие поля и типы реально используются.
3. **Просмотр структуры из приложения:**  
   `php artisan lmb:db-schema --table=_reference122x1`  
   и т.д. — видно только физические имена колонок (`_fld3184`, `_fld41084`), смысл которых уточняется по выгрузке метаданных.

## 7. Команды и конфиг для работы со структурой

| Действие | Команда / конфиг |
|----------|-------------------|
| Список таблиц по префиксу | `php artisan lmb:db-tables --prefix=_reference --no-vt` (только справочники без табличных частей). Опции: `--count` (подсчёт записей), `--limit=N`. |
| Список таблиц и колонок | `php artisan lmb:db-schema` (опции: `--table=NAME`, `--limit=N`). |
| Подключение к БД 1С | `.env`: `LMB_DB_DRIVER=pgsql`, `LMB_DB_HOST`, `LMB_DB_DATABASE`, `LMB_DB_USERNAME`, `LMB_DB_PASSWORD`. |
| Таблица контрагентов (физлица) | `config/services.php` → `lmb_1c_contragent_sync.table` (по умолчанию `_reference122x1`). |
| Синхронизация клиентов | `php artisan lmb:sync-contragents` (и `--dry-run`). |
| Удаление «банков» из клиентов | `php artisan lmb:clean-bank-clients` (опции: `--dry-run`, `--force`). |

## 8. План дальнейшей синхронизации

| Данные | Источник в 1С | Наша сущность | Статус |
|--------|----------------|---------------|--------|
| Физлица (ФИО, телефон, паспорт) | `_reference122x1` | `clients` | Реализовано |
| Залоги (договоры залога) | Документ «Залог» (_document???) | `pawn_contracts` | Реализовано: `lmb:sync-pawn-contracts`, см. `docs/LMB_SYNC_PAWN.md` |
| Выкупы / реализация | Документы выкупа/реализации | Соответствующие сущности | Аналогично |
| Номенклатура (товары) | Справочник номенклатуры | Каталог товаров (если нужен) | По метаданным |
| Остатки по складам | Регистры накопления | Остатки в приложении | По метаданным |

Рекомендуемый порядок: получить выгрузку метаданных из 1С → определить точные имена таблиц документов «Залог» и «Выкуп» → описать их колонки и связи с `_reference122x1` в этом документе → реализовать синхронизацию залогов/выкупов.

**Остатки из регистров накопления:** команда `lmb:1c-balances-discovery` выводит список таблиц `_accumrg*` и структуру; синхронизация в таблицу `lmb_register_balances` — см. **`docs/LMB_1C_BALANCES_SYNC.md`**.

**Полная интеграция (залог, скупка, вещи):** пошаговая настройка и команды — **`docs/LMB_INTEGRATION_FULL.md`**. Найти таблицу документа залога: `php artisan lmb:discover-1c-documents --count --exclude-known --with-vt`.

**Маппинг по реальной выгрузке 1С:** в документе **`docs/LMB_1C_REPORT_AND_MAPPING.md`** описана структура отчёта из 1С (операции по залогу и скупка), колонки (дата, номер, точка, клиент, наименование товара, сумма), список точек для сопоставления с `stores` и схема связей таблиц 1С и приложения для интеграции.

## 9. Примеры запросов (PostgreSQL)

Подключение от приложения: соединение `lmb_1c_pgsql`.

**Количество физлиц (без пометки):**
```sql
SELECT COUNT(*) FROM public._reference122x1 WHERE NOT _marked;
```

**Выборка контрагентов с паспортом и телефоном:**
```sql
SELECT encode(_idrref, 'hex') AS uid,
       _description,
       _fld3178 AS full_name,
       _fld41084 AS phone,
       _fld3184 AS passport_series,
       _fld3185 AS passport_number
FROM public._reference122x1
WHERE NOT _marked AND (_fld41084 IS NOT NULL AND _fld41084 <> '');
```

**Поиск документов с ссылкой на контрагента** (после того как по метаданным станет известен номер документа и имя поля): подставить в условие `_fldXXXXrref = decode('hex_uid', 'hex')` для нужного `user_uid`.
