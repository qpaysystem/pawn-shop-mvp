## 3. Таблицы 1С: что уже известно

### Документ залога (текущая настройка)

| Таблица | Записей | Назначение |
|---------|---------|------------|
| **_document382** | 4 | Документ залога (в .env: LMB_1C_PAWN_DOCUMENT_TABLE). В выборке: _number = «0000-000001», не формат ЕЛЗ-XXXXX-ДК. |
| **_document382_vt9463** | 7 | Табличная часть: номенклатура (_fld9466rref), краткое описание (_fld9467, mvarchar 20), сумма по строке (_fld40394). |

**Вывод:** в _document382 всего 4 документа; отчёт инвентарки (3299 строк) формируется **не** из этой таблицы. Возможные источники:
- другой документ с номерами вида ЕЛЗ-XXXXX-ДК (например «Залоговый билет» или «Операция по залогу» в другой конфигурации);
- регистр накопления (остатки залоговых вещей);
- представление/обработка 1С, объединяющая несколько документов и серий номеров.

### Документы с большим числом записей (кандидаты под «операции по залогу»)

По результатам `php artisan lmb:db-tables --prefix=_document --no-vt --count`:

| Таблица | Записей | Примечание |
|---------|---------|------------|
| _document41615x1 | 321 813 | Максимум записей — возможный журнал операций/залоговых билетов. |
| _document39540x1 | 242 022 | |
| _document41638x1 | 15 329 | |
| _document450 | 8 911 | |
| _document42615x1 | 5 093 | |
| _document41736x1 | 1 871 | |
| _document41694x1 | 1 245 | Близко к 3299 при фильтрации (например по дате/складу). |
| _document389x1 | 168 | Скупка (уже используется). |
| _document382 | 4 | Текущая таблица залога. |

В БД номера хранятся в «сыром» виде (например _document41615x1: «000324760», _document41694x1: «000000149»). Формат **ЕЛЗ-10198-ДК** в отчёте собирается в 1С из серии (ЕЛЗ), номера и суффикса филиала (ДК).

Рекомендация: выгрузить из 1С метаданные (см. `docs/1c_export_metadata_to_file.txt`) и определить, какой объект метаданных соответствует документу с номерами ЕЛЗ-*-ДК и колонками «Филиал», «Документ», «Наименование», «Поступление», «Срок», «Стоимость», «Клиент», «Проба», «Вес». Затем сопоставить его с одной из таблиц _document* (например _document41615x1 или _document41694x1).

### Регистры накопления (остатки)

Команда `php artisan lmb:1c-balances-discovery` выводит таблицы _accumrg* (около 20). Остатки залоговых вещей по складам могут храниться в одном из этих регистров. Для идентификации нужна выгрузка метаданных (имя регистра «Залоговые вещи» / «Остатки» и измерения: склад, номенклатура, документ и т.д.).

### Справочник номенклатуры

Наименование, проба, вес в отчёте могут браться из справочника номенклатуры 1С. В ТЧ залога _document382_vt9463 есть _fld9466rref (ссылка на номенклатуру). Таблица справочника — _reference* (конкретный номер по метаданным). Поиск: `php artisan lmb:db-tables --prefix=_reference --no-vt --count --limit=50`.

## 4. Перенести все договоры залога из 1С в нашу базу

**Да.** Перенос уже реализован: команда синхронизации читает документ залога из БД 1С и создаёт/обновляет записи в `pawn_contracts`, `items` и при необходимости в `clients`.

- **С текущей настройкой** (в .env указана таблица _document382): переносятся все договоры из этой таблицы. В вашей базе 1С в _document382 только 4 документа → к нам попадёт 4 договора (7 позиций по табличной части).
- **Чтобы перенести все договоры** (как в отчёте инвентарки — тысячи строк): нужно указать в .env **ту таблицу 1С**, где реально хранятся все операции по залогу (кандидаты: _document41615x1, _document41694x1 и др., см. раздел 3 выше), и задать для неё имена колонок (контрагент, дата, сумма, табличная часть). После этого та же команда перенесёт все договоры из этой таблицы.

**Команда переноса (все залоги из выбранной таблицы, без подтверждения):**
```bash
php artisan lmb:sync-pawn-contracts --all --force
```

Без `--all` переносятся только действующие (дата окончания ≥ сегодня). Перед первым запуском желательно синхронизировать контрагентов: `php artisan lmb:sync-contragents`. При отсутствии контрагента в нашей базе можно включить создание заглушки: `LMB_1C_PAWN_CREATE_PLACEHOLDER_CLIENT=true` в .env.

## 5. Действия для переноса данных в нашу базу (детали)

### Вариант A: Синхронизация из БД 1С (предпочтительно)

1. **Определить источник отчёта инвентарки в 1С**  
   По метаданным или по перебору: документ или регистр, в которых есть номера ЕЛЗ-*-ДК, филиал, клиент, наименование, проба, вес, сумма, даты.

2. **Расширить маппинг в приложении**  
   - Если источник — документ: задать в config/services.php таблицу и колонки (аналог lmb_1c_pawn_sync), включая склад/филиал, номенклатуру, проба/вес (если есть в документе или в справочнике).  
   - Если источник — регистр остатков: использовать или доработать механику из `docs/LMB_1C_BALANCES_SYNC.md`, замапить измерения на store_id, item, contract.

3. **Маппинг «Филиал» → stores**  
   Создать в конфиге или в БД соответствие наименований точек из отчёта (Дуси Ковальчук 266/2, Горский 1, …) и stores.id.

4. **Запуск синхронизации**  
   После настройки: синхронизация контрагентов (`lmb:sync-contragents`), затем залогов (`lmb:sync-pawn-contracts`) и при необходимости скупки (`lmb:sync-purchase-contracts`). Для залогов при отсутствии контрагента в нашей базе можно включить `LMB_1C_PAWN_CREATE_PLACEHOLDER_CLIENT=true`.

### Вариант B: Импорт из файла инвентарки (XLS)

Реализована команда **`lmb:import-inventory`** — создаёт по файлу инвентаризации магазины, клиентов (физлиц), товары, договоры залога и договоры скупки.

1. **Парсинг структуры:** `php artisan lmb:parse-inventory-xls /path/to/Инвентарка.xls` — вывод колонок и примеров строк.
2. **Импорт в БД:**  
   `php artisan lmb:import-inventory /path/to/Инвентарка.xls`  
   Опции: `--dry-run` (только подсчёт), `--force` (без подтверждения), `--sheet=0`, `--header-row=4`.
3. По Col1 создаются магазины (Филиал: Горки 1, Мичурина, Дуси Ковальчук 266/2 и т.д.), по Col19+Col21 — клиенты, по Col2/5/7/8/10/12/14/15 — залоги и товары. Строки с признаком скупки/распродажи (Col26 или номер документа) создают договоры скупки.

## 6. Донастройка синхронизации из БД 1С

### 6.1. Склад/филиал (store_id)

Если в документе залога в 1С есть реквизит-ссылка на склад/филиал:

1. Узнать имя колонки: в схеме документа (например `php artisan lmb:db-schema --table=_document382`) найти поле типа `*rref`, по метаданным 1С — «Склад» или «Филиал». Пример: `_fld9450rref`.
2. В `.env` задать:
   ```env
   LMB_1C_PAWN_STORE_COLUMN=_fld9450rref
   ```
3. Маппинг 1С → наш магазин:
   - **Через Cursor/терминал:** выполнить `php artisan lmb:list-pawn-store-refs --column=_fld9450rref` (или без `--column`, если задан `LMB_1C_PAWN_STORE_COLUMN`). Команда выведет список кодов складов. Далее:
     - `--assign` — сопоставить по порядку с нашими магазинами и вывести готовую строку для `LMB_1C_PAWN_STORE_MAPPING` в .env;
     - `--json` — шаблон JSON для ручного заполнения store_id.
   - **Вариант A:** в таблице `stores` заполнить поле `lmb_store_uid` (hex из вывода команды).
   - **Вариант B:** в `.env` задать `LMB_1C_PAWN_STORE_MAPPING={"hex_uid_1c": store_id}`, например `{"80e10050569c4e3011e7365a18a6185b":1}`.

Миграция `2026_03_11_120001_add_lmb_store_uid_to_stores_table.php` добавляет в `stores` колонку `lmb_store_uid`.

### 6.2. Сумма займа и товар

- При создании позиции залога из 1С для товара (`items`) автоматически проставляется `initial_price` = сумма займа по строке/документу.
- Наименование товара берётся из табличной части (`vt_description_column`) или формируется как «Залог 1С» + номер документа.

### 6.3. Контрагент и заглушки

- Включение `LMB_1C_PAWN_CREATE_PLACEHOLDER_CLIENT=true` создаёт клиента «Клиент 1С (залог)» при отсутствии контрагента в нашей базе, чтобы залоги не пропускались.
